26. heuristic_ik() — MEANINGLESS FAKE IK FALLBACK  [DONE]
================================================================================
Where: streams.py lines 268-285
What:  When robot_id is None (no PyBullet robot loaded),
       _compute_sensing_ik() falls back to _heuristic_ik() which
       produces "plausible-looking" joint configs by:
         offset = (ee_pos - [0.5, 0, 0.5]) * 0.5
         perturb home config by offset * 0.3

       These are NOT inverse kinematics — they are random perturbations
       of the home pose. The resulting configurations have no relationship
       to the requested end-effector position. They exist only to let
       tests run without a robot, but they could silently corrupt
       results if used when a robot IS expected.

       from _heuristic_ik() are marked is_heuristic=True so downstream
       code can detect and reject them when real IK is expected.
       (2) BoxelStreams.__init__ logs a WARNING when robot_id is None,
       alerting early that all IK will use the heuristic fallback.
       (3) _heuristic_ik() logs a WARNING on every call with the
       requested ee_pos, making it impossible to miss silently.
       (4) Docstring rewritten to honestly describe what the function
       does (arbitrary perturbation, NOT inverse kinematics).

47. PUSH STREAM DOMAIN USES is_object AS HACK TO BIND ?b_from  [DONE]
================================================================================
Where: pddl/stream.pddl, pddlstream_planner.py, streams.py
What:  The push stream's domain condition was:
         (and (is_object ?obj) (is_object ?b_from))
       ?b_from is a LOCATION (the boxel where the object sits), not an
       object. Using is_object exploited the coincidence that occluders
       initially sit at their own object-typed boxels. This caused:
         - Semantically wrong: a location is not "an object."
         - Over-instantiates the stream for all object├ùobject pairs.
         - Breaks if an occluder is pushed to a non-object boxel and
           needs to be pushed again from there.

Fixed: Introduced a static predicate (occluder_at ?occ ?b) that mirrors
       each occluder's current position. The stream domain is now:
         (and (is_object ?obj) (occluder_at ?obj ?b_from))

       Why not obj_at_boxel directly? PDDLStream strips fluent predicates
       (those in action effects) from stream domains. Using obj_at_boxel
       causes KeyError: '?b_from' because the Instantiator can't bind it
       after stripping.

       Why not :fluents (obj_at_boxel)? compile_fluent_streams() enforces
       that fluent stream outputs cannot appear in action effects. Push
       stream outputs (?b_to, ?q_start, ?q_end, ?traj) all appear in
       push action effects — blocked by design.

       The static predicate approach:
       (1) domain_pddlstream.pddl: added (occluder_at ?o ?b) predicate.
       (2) pddlstream_planner.py _build_init(): emits (occluder_at occ loc)
           for each occluder — at original boxel or at moved destination
           (from #48's moved_occluders dict).
       (3) stream.pddl: push stream domain changed to (occluder_at ?obj ?b_from).
       (4) Removed b_from != occluder_id guard from generators in both
           pddlstream_planner.py and streams.py — the domain now guarantees
           only valid (occluder, location) pairs reach the generator.

       Result: semantically correct binding, no over-instantiation
       (evaluations dropped from 237 to 183 in test), works with moved
       occluder destinations from #48. Limitation: occluder_at is static
       per planning call, so re-pushing the same occluder twice within
       a single plan is not supported — the replanning loop handles this.

49. STREAM IK FAILS ON REPLAN — PLANNER TIMES OUT  [DONE]
================================================================================
Where: streams.py (_pybullet_ik), test_full_pipeline.py (replanning loop)
What:  When the target is NOT behind the first pushed occluder, Plan #1
       succeeds (pushes, senses, finds nothing). Plan #2 then times out
       (60s) because stream sampling produces almost nothing — evaluations
       only increase from 183 to 185 (2 new atoms vs 15 in Plan #1).

       Root cause has two parts:

       (1) SEED DEPENDENCY: PyBullet's calculateInverseKinematics() uses
       the robot's CURRENT joint state as its iterative solver seed.
       After Plan #1 moves the robot (move_robot_to_pos in execution),
       the joints are far from REST_POSES. On Plan #2, every IK call
       in _pybullet_ik() starts from this displaced configuration,
       converging to solutions that violate joint limits (filtered out)
       or not converging at all. Plan #1 works because the robot starts
       at REST_POSES (a good seed).

       (2) NO NULL-SPACE PARAMETERS: _pybullet_ik() in streams.py calls
       calculateInverseKinematics without lowerLimits, upperLimits,
       jointRanges, or restPoses. Compare with compute_ik() in
       robot_utils.py (used by execution), which provides all four —
       making it much more robust. The stream IK is the simple variant;
       the execution IK is the null-space variant. They should match.

       Evidence: Plan #2 output shows sample_time: 0.028s (generators
       fast but empty), search_time: 59.97s (planner retrying with no
       new atoms). 4 skeletons found (one per occluder) but 0 solutions.

Fixed: (a) _pybullet_ik() now saves the robot's current joint state,
       resets all arm joints to REST_POSES (providing a consistent seed),
       runs IK, then restores the original state in a finally block.
       This makes stream IK results independent of where plan execution
       left the robot.
       (b) calculateInverseKinematics call now includes lowerLimits,
       upperLimits, jointRanges, and restPoses — matching compute_ik()
       in robot_utils.py. The null-space parameters bias solutions toward

61. SENSE ACTION DEVIATES FROM PROPOSAL — NO CONDITIONAL EFFECTS  [ACCEPTED]
================================================================================
Where: pddl/domain_pddlstream.pddl (sense action),
       test_full_pipeline.py (sense execution + replan loop),
       proposal Section 4.4.2 (Listing 2)

What:  The PDDL sense action unconditionally adds (obj_at_boxel ?o ?region)
       and (obj_pose_known ?o).  It assumes the target will always be found
       ("optimistic").  The proposal's conditional design with
       stream_get_sensing_outcome and (when found/not_found ...) branches
       is not implemented.  Instead, the execution loop detects sensing
       failures and replans with updated belief (known_empty_shadows).

Status: ACCEPTED DEVIATION — documented in PDDL domain file and code.

Justification:
       (a) PDDLStream + FastDownward cannot do contingent planning.
           Branching on observation outcomes requires a contingent planner
           (POND, CLG), which is outside PDDLStream's architecture.  The
           proposal's conditional-effect design cannot be implemented
           within the chosen planning framework without fundamental changes.
       (b) Optimistic planning with replanning on failure is a standard
           TAMP pattern under partial observability (Garrett et al., 2020;
           Kaelbling & Lozano-Pérez, 2013).  PDDLStream's adaptive
           algorithm is itself built around optimistic assumptions.
       (c) For N shadow candidates, reactive replanning converges in at
           most N replan cycles (one per empty shadow).  Belief state
           propagates correctly: known_empty_shadows carry over, so each
           replan searches strictly fewer candidates.
       (d) The execution loop bounds the search: max_replans = 4*N + 1.

Limitation:
       The planner cannot optimize search ORDER within a single plan.
       A contingent planner could reason about most-likely-first strategies.
       For the current uniform-prior tabletop scenario this does not affect
       completeness or practical efficiency.

Verified: 2026-03-22 — domain PDDL has optimistic sense; execution loop
       has tri-state replanning with known_empty_shadows propagation.

       (free/occupied/shadow + re-merge), which subsumes this issue.

       functionally safe for pick-and-place: the occluder is physically

#5. PLACEMENT RE-BLOCKS VIEW CORRIDOR (DESTINATION IN CAMERA LOS)  [DONE]

(Moved from CODEBASE_AUDIT.txt 2026-04-06.  The open-audit body had been
 mislabeled “#2”; summary table used #5 = old #107.)

Priority: TIER 1 — Needs design decision before implementation
Where:    domain_pddlstream.pddl (place action),
          pddlstream_planner.py (_build_init: boxel_fits facts),
          streams.py (BoxelStreams.test_boxel_fits)
Depends:  None (but benefits from #4)
Effort:   2-3 days (after design is decided)

What:  The place action requires only that the destination is free and
       large enough (is_free_space + boxel_fits; see policy above).  It does NOT check
       whether placing the occluder at the destination re-blocks the
       view corridor the plan just cleared.

       Example: occluder blocks shadow_007.  Planner places occluder
       at a free boxel that is ALSO in the camera LOS cone to
       shadow_007.  After place, view is still blocked — the plan
       accomplishes nothing.

       Two sub-problems:
       (a) Destination re-blocks the SAME shadow (primary case).
       (b) Destination blocks ANY shadow (stricter, may be too
           conservative).

Fix:   ONE COMMIT — candidate approaches (design decision needed):

       OPTION A — Test stream: test-place-clears-view(?o ?dest ?shadow)
       returns True iff placing ?o at ?dest does NOT block ?shadow.
       Add as precondition on place.

       OPTION B — Pre-computed safe set: at problem build time, for
       each free boxel compute whether it lies in any LOS cone.  Emit
       (no_view_block ?b) facts.  Place requires (no_view_block ?b).

       OPTION C — Explicit placement-region set: pre-approved "safe
       placement" boxels (outside all camera cones) as init facts.

Fixed: placement_blocking / blocks_view_at emission in
       pddlstream_planner.py (_build_init) and derived visibility in the
       domain (audit #5 in codebase comments).

Related: #4, #6

#19. MID-AIR FREE BOXELS USED AS PLACEMENT — GRAVITY DROPS INTO SHADOW  [DONE]
================================================================================
(Moved from CODEBASE_AUDIT.txt 2026-04-06.)

Priority: TIER 0 — Can be fixed independently, simple guard
Where:    pddlstream_planner.py (_build_init: is_free_space emission),
          test_full_pipeline.py (execute_place, gravity settle step)
Depends:  None
Effort:   0.5 day

What:  During the 2026-04-05 run (Plan #3), the planner chose free_012 as
       the drop destination for obj_004 (occluder_1), which was blocking
       shadow_000.  After placement, sensing shadow_000 reported:

         "View to shadow_000 still blocked (69% rays hit occluder)"

       Investigation of boxel_data.json reveals that free_012 and
       shadow_000 do NOT overlap in 3D — the octree is correct:

         shadow_000: min=[0.053, 0.238, 0.325] max=[0.147, 0.497, 0.400]
         free_012:   min=[-0.15, 0.250, 0.450] max=[0.350, 0.500, 0.575]

       Their Z ranges are disjoint (shadow max Z=0.400 < free min Z=0.450).
       free_012 is a mid-air boxel (on_surface: null) floating ABOVE
       shadow_000.

       However, their X-Y projections fully overlap:
         X: shadow [0.053, 0.147]  ⊂  free [-0.15, 0.350]
         Y: shadow [0.238, 0.497]  ⊂  free [0.250, 0.500]

       The placement execution chain causes the problem:
         1. boxel_centers["free_012"] = [0.10, 0.375, 0.5125]  (mid-air)
         2. compute_kin targets EE at boxel center + grasp offset
         3. Robot moves there, opens gripper, releases constraint
         4. p.stepSimulation() runs 30 times — GRAVITY drops the object
            from Z=0.51 to the TABLE at Z≈0.363
         5. At (X=0.10, Y=0.375) on the table surface, the object is
            dead center inside shadow_000 (X: 0.053-0.147, Y: 0.238-0.497)

       The boxelization is correct — a free boxel CAN exist above a
       shadow region in 3D.  The bug is that the planner treats mid-air
       boxels as valid placement targets, ignoring that gravity will
       project the object down to the table surface, where its X-Y
       footprint collides with the shadow region below.

       This also explains why free_012 was reused in Plan #1 (to place
       obj_006) — the planner happily picks mid-air boxels.

Fix:   ONE COMMIT — PDDL-level fix using an (on_surface) fluent:

       The fix belongs in the PDDL domain, not hardcoded in Python.
       The place action should only allow placement in boxels that are
       both known-free AND on a support surface.

       (a) domain_pddlstream.pddl:
           - Add predicate: (on_surface ?b)
           - Add precondition to the place action:
               (on_surface ?b)
             alongside the existing (is_free_space ?b) and
             (boxel_fits ?o ?b).
           The place action then requires:
             (is_free_space ?b)   — known empty
             (on_surface ?b)      — resting on a support surface
             (boxel_fits ?o ?b)   — large enough for the object

       (b) pddlstream_planner.py (_build_init):
           - Emit (on_surface ?b) for every boxel where
             boxel.on_surface is not None.
           This is a fluent, not static: after re-boxelization (#4)
           the on_surface status of boxels may change.

       (c) No stream.pddl changes needed — on_surface is a simple
           init fact, not a stream-certified predicate.

       This eliminates 12 of 44 free boxels (those with on_surface=null)
       from placement consideration.  Mid-air boxels remain in the
       registry for spatial reasoning but cannot be placement targets.

Fixed: (on_surface ?b) added to domain and _build_init as described.

Related: #5 (placement re-blocks view corridor — X-Y overlap variant)
         #4 (stale is_free_space facts after prior placements)

#22. PLACE ALLOWS MULTIPLE OBJECTS IN THE SAME FREE BOXEL  [DONE]
================================================================================
(Moved from CODEBASE_AUDIT.txt 2026-04-06.)

Priority: TIER 0 — Quick PDDL fix
Where:    domain_pddlstream.pddl (place action effects)
Depends:  None
Effort:   ~0.5 day

What:  The place action's precondition checks (is_free_space ?b).  That
       fact was only removed from the planning state implicitly if the
       planner never needed to reason about it again — but (is_free_space ?b)
       is not deleted when (obj_at_boxel ?o ?b) is added.  A second place
       in the same plan could still satisfy (is_free_space ?b) and stack
       another object on the same boxel.

       Observed: planner places two occluders at the same free boxel in
       the same plan.  Physically the objects overlap in PyBullet and
       one or both may be knocked off the table by the physics settle.

Fix:   Add (not (is_free_space ?b)) to the place action :effect (same
       pattern as (not (holding ?o))).

Fixed: 2026-04-06 — effect (not (is_free_space ?b)) on place.

       Rationale vs. derived (boxel_occupied): avoids an extra derived
       predicate; “slot no longer empty free-space” is tracked on the
       fluent place already consumes.

       Note:  _build_init still emits (is_free_space ?b) for every registry
       FREE_SPACE boxel on each replan, even when moved_occluders already
       places an object there (#4 / init hygiene).  #22 addressed intra-plan
       duplicates; tightening init or a derived occupancy check would close
       cross-replan double-placement if that appears in runs.

Related: #4 (re-boxelization / init consistency for is_free_space)
