       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET GUI FREEZES DURING PLANNING — THIS IS NORMAL
       The RenderingLock (robot_utils.py) disables COV_ENABLE_RENDERING
       for the entire planning phase (test_full_pipeline.py ~line 346,
       audit #60).  While planning runs (often 30-130 s), the PyBullet
       window appears frozen — no zoom, pan, or rotation works.
       The GUI unfreezes when planning finishes.
       If planning takes very long, WSLg's Weston compositor may drop
       the X11 connection, crashing the process with:
         XIO: fatal IO error 11 (Resource temporarily unavailable) on X server ":0"
       Fix: restart WSL (wsl --shutdown from PowerShell, then reopen
       terminal).  Keyboard camera controls (R/F for zoom, WASD for
       pan) also do not work while rendering is locked.

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

#1. EXECUTION SUB-MOVES ARE COLLISION-BLIND
================================================================================
Priority: TIER 0 — Start here (most self-contained)
Where:    test_full_pipeline.py (execute_pick ~line 782, execute_place ~line 894)
          robot_utils.py (move_robot_smooth)
Depends:  None
Effort:   1-2 days

What:  The sub-moves inside execute_pick (approach, descend-to-contact,
       lift) and execute_place (approach, descend, retreat) all use
       move_robot_smooth(), which is purely linear joint-space
       interpolation with NO collision checking.

       The PDDL planner calls plan_motion() (full RRT-Connect collision
       checking) only for the symbolic move action.  Everything inside
       execute_pick/execute_place runs blind at execution time.

       Consequence: the robot collides with other objects during
       approach/retreat.  The hardcoded heights (approach 0.10 m,
       lift/retreat 0.25 m) are never validated against the scene.
       This is the observed failure: "robot struggles with move to
       place path."

Fix:   ONE COMMIT — replace move_robot_smooth() calls with
       collision-aware motion:
       (a) In execute_pick: for the approach and lift segments, call
           BoxelStreams.plan_motion() or is_path_collision_free() to
           get a collision-free trajectory, then follow its waypoints.
       (b) In execute_place: same for approach and retreat segments.
       (c) Pass ignored_body_ids (grasped object during pick approach;
           none during retreat) matching what plan_motion already does
           for the PDDL move action.
       The descend-to-contact segment can remain linear (short vertical
       move in collision with the target object by design).

       Reuses existing RRT/collision infrastructure with minimal new
       code and NO PDDL domain changes.

Related: #7/#8 (full PDDL decomposition is the long-term fix)

#2. BOXEL MERGE CONDITION INCOMPLETE
================================================================================
Priority: TIER 0 — Prerequisite for #4 and #6
Where:    cell_merger.py, boxel_data.py
Depends:  None
Effort:   2-3 days

What:  The current merge logic only checks whether two adjacent free
       boxels produce a CONVEX result.  Per the supervisor meeting
       (2026-03-22), two additional conditions are required:

       Two adjacent boxels should be merged if and only if:
         (a) Convex result (existing check — may need revisiting per
             supervisor's observation that "a merging operation
             definitely does not produce convex boxes"), AND
         (b) IDENTICAL logical properties — they block the view to
             exactly the same set of target objects (same
             blocks_view_at facts), AND
         (c) Same observability status — BOTH currently observable OR
             BOTH completely in a shadow region.

       Merge ORDER: back-to-front (far from camera first), per
       supervisor guidance.

       Consequence of not merging correctly: boxels with different
       logical properties get merged, producing boxels whose symbolic
       meaning is undefined — PDDL state cannot correctly represent
       what they block.

Fix:   ONE COMMIT — in cell_merger.py merge_free_space():
       (a) Accept shadow/blocking information as parameters (which
           targets does each boxel's camera corridor obstruct).
       (b) Before merging two candidates, check:
           - Both observable or both in shadow
           - Both have identical blocks_view_at target sets
       (c) Process candidates back-to-front.
       (d) Merged boxel inherits the common property set.

Related: #4 (re-boxelization depends on correct merge condition),
         #6 (multi-boxel placement depends on correct merging)

#3. SENSE ACTION HIDDEN MOVE — NO at_config PRECONDITION
================================================================================
Priority: TIER 0 — Quick PDDL consistency fix
Where:    domain_pddlstream.pddl (sense action, ~line 128),
          test_full_pipeline.py (sense handler, ~line 425)
Depends:  None
Effort:   1-2 days

What:  The sense handler secretly moves the robot to home
       configuration before raycasting (workaround for old #79: arm
       blocking camera).  The PDDL sense action has NO at_config
       precondition — the planner does not know sensing requires the
       arm to be out of the way.

       This causes PDDL state drift: after sense, the robot is
       physically at home but PDDL still records (at_config ?q) from
       the last move action.  If the planner chains sense -> pick,
       the pick starts from the wrong config in the symbolic model.

Fix:   ONE COMMIT — choose one approach:

       OPTION A (minimal): Add (at_config ?q_home) as a precondition
       to the sense action in domain_pddlstream.pddl.  Emit
       (at_config q_home) in _build_init() when the robot starts at
       home.  The planner will generate move -> sense sequences when
       the robot is not at home.  Remove the hidden move from the
       sense handler (the planner now handles it explicitly).

       OPTION B (pragmatic): Add a prepare_sense action that moves
       the robot to home.  The sense action requires (at_config
       ?q_home) as a precondition.  prepare_sense's effect sets
       at_config to q_home.  More explicit but adds a PDDL action.

       OPTION A is recommended — simpler, and move already exists.

Related: old #79 [DONE] (arm blocks sensing), old #86 [DONE] (config
         tracking on replan)

#4. POST-ACTION RE-BOXELIZATION — STALE PARTITION
================================================================================
Priority: TIER 1
Where:    test_full_pipeline.py (execution loop),
          boxel_data.py (BoxelRegistry),
          cell_merger.py
Depends:  #2 (merge condition must be correct before re-merging)
Effort:   3-5 days

What:  After a pick-and-place action, the boxel partition is NOT
       recomputed.  Stale data includes:
       (a) OCCUPIED boxels that are now FREE (object picked up)
       (b) FREE boxels that are now OCCUPIED (object placed there)
       (c) Shadow regions that changed (occluder moved)
       (d) Merge structure is stale (needs re-merge with #2 rules)

       The planner operates on this stale partition, causing it to:
       - Try placing in no-longer-free boxels
       - Miss newly available free space
       - Plan sense actions for shadows that no longer exist

Fix:   ONE COMMIT — implement full re-boxelization (naive first):
       (a) After each scene-changing action (place), re-evaluate every
           boxel's occupancy status against current PyBullet state.
       (b) Recompute shadow regions for relocated occluders.
       (c) Re-run the merge pass using the correct merge condition
           from #2.
       (d) Rebuild shadow_occluder_map.
       (e) Feed updated BoxelRegistry into _build_init() for next
           planning cycle.

       CRITICAL: observation history (which boxels have been
       sensed / known-empty) MUST survive re-boxelization.

       Incremental recomputation is in FOR LATER.

Related: old #44 [ACCEPTED] (stale shadow AABBs — subsumed by this),
         old #73 [DONE] (stale shadow_occluder_map — extended by this)

#5. PLACEMENT DESTINATION MAY RE-BLOCK THE CLEARED VIEW CORRIDOR
================================================================================
Priority: TIER 1 — Needs design decision before implementation
Where:    domain_pddlstream.pddl (place action),
          pddlstream_planner.py (_build_init, _get_stream_map),
          streams.py (BoxelStreams)
Depends:  None (but benefits from #2 and #4)
Effort:   2-3 days (after design is decided)

What:  The place action requires only that the destination is free and
       large enough (is_free_space + boxel_fits).  It does NOT check
       whether placing the occluder at the destination re-blocks the
       view corridor the plan just cleared.

       Example: occluder blocks shadow_007.  Planner places occluder
       at a free boxel that is ALSO in the camera LOS cone to
       shadow_007.  After place, view is still blocked — the plan
       accomplishes nothing.

       Two sub-problems:
       (a) Destination re-blocks the SAME shadow (primary case).
       (b) Destination blocks ANY shadow (stricter, may be too
           conservative).

Fix:   ONE COMMIT — candidate approaches (design decision needed):

       OPTION A — Test stream: test-place-clears-view(?o ?dest ?shadow)
       returns True iff placing ?o at ?dest does NOT block ?shadow.
       Add as precondition on place.

       OPTION B — Pre-computed safe set: at problem build time, for
       each free boxel compute whether it lies in any LOS cone.  Emit
       (no_view_block ?b) facts.  Place requires (no_view_block ?b).

       OPTION C — Explicit placement-region set: pre-approved "safe
       placement" boxels (outside all camera cones) as init facts.

Related: #2, #4, #6

#6. MULTI-BOXEL FREE SPACE PLACEMENT
================================================================================
Priority: TIER 1 — Depends on #2 for correct merging
Where:    domain_pddlstream.pddl, stream.pddl, streams.py,
          pddlstream_planner.py
Depends:  #2 (correct merge condition reduces need for multi-boxel)
Effort:   ~1 week

What:  The place action targets a SINGLE boxel.  If no single free
       boxel is large enough, the planner cannot reason about placing
       across multiple adjacent free boxels.

       After merging with correct #2 condition, free boxels with
       identical properties merge into large regions, largely
       mitigating this.  Multi-boxel placement is needed only for
       objects at occlusion boundaries where boxels stay small.

Fix:   ONE COMMIT — multi-variant place actions (agreed in meeting):
       (a) Add place-in-2-boxels action to domain_pddlstream.pddl:
           precondition requires both target boxels are adjacent
           and FREE; PDDLStream certifier verifies object fits in
           the union.
       (b) Add corresponding stream in stream.pddl.
       (c) Implement adjacency + union-fitness test in streams.py.
       (d) Optionally generalise to place-in-N-boxels.

       If #2 merging works well, this may become unnecessary for
       the thesis scenarios.  Implement only if testing reveals
       placement failures due to boxel size.

Related: #2 (merge condition)

#7. PICK ACTION — PDDL/PYTHON SCOPE MISMATCH
================================================================================
Priority: TIER 2 — PDDL redesign, optional for thesis
Where:    domain_pddlstream.pddl (pick action, ~line 178),
          stream.pddl, streams.py, test_full_pipeline.py (execute_pick)
Depends:  None (but #1 provides a lighter-weight mitigation)
Effort:   ~1 week

What:  The PDDL pick action is modeled as an instantaneous single-
       config operation.  The Python execute_pick() is a 6-step
       choreographed sequence:
         1. open_gripper
         2. move to approach (IK solved at execution time)
         3. descend to contact (plan config)
         4. close_gripper
         5. createConstraint (weld object to gripper)
         6. lift (IK solved at execution time)

       Hidden sub-actions the planner cannot reason about:
         3 moves, 3 IK solves, 1 open_gripper, 1 close_gripper,
         1 createConstraint

       Consequences:
       - Planner cannot detect collisions on approach/retreat (#1
         provides a lighter-weight fix for this)
       - Execution-time IK solves can fail (handled by abort-and-
         replan, but planner has no awareness)
       - Post-pick config differs from PDDL's ?q (accepted
         simplification — old #62)

Fix:   ONE COMMIT — decompose pick into PDDL sub-actions:
       (a) approach_pick(?o ?b ?g ?q_approach)
       (b) grasp(?o ?g)
       (c) lift(?o ?q_lift)

       Each sub-action has its own preconditions and effects,
       including at_config updates.  Stream evaluations ~3x per pick.

       NOTE: #1 (collision-aware sub-moves) is the pragmatic fix.
       This full decomposition is architecturally correct but optional
       for thesis scope.

Related: #1 (lighter-weight collision fix), #8 (same pattern for place)

#8. PLACE ACTION — PDDL/PYTHON SCOPE MISMATCH
================================================================================
Priority: TIER 2 — PDDL redesign, optional for thesis
Where:    domain_pddlstream.pddl (place action, ~line 205),
          stream.pddl, streams.py,
          test_full_pipeline.py (execute_place)
Depends:  None (but #1 provides a lighter-weight mitigation)
Effort:   ~1 week

What:  Same pattern as #7.  The PDDL place action is instantaneous
       single-config.  The Python execute_place() is a 6-step
       sequence:
         1. move to approach (IK solved at execution time)
         2. descend to contact (plan config)
         3. open_gripper
         4. removeConstraint
         5. 30 stepSimulation (settling)
         6. retreat (IK solved at execution time)

       Hidden sub-actions:
         3 moves, 3 IK solves, 1 open_gripper, 1 removeConstraint,
         30 stepSimulation

Fix:   ONE COMMIT — decompose place into PDDL sub-actions:
       (a) approach_place(?o ?b ?g ?q_approach)
       (b) release(?o ?b)
       (c) retreat(?q_retreat)

       Same reasoning as #7: #1 is the pragmatic fix; this is the
       architecturally correct long-term solution.

Related: #1, #7

#13. RECURSIVE OBJECT DISCOVERY IN FREE SPACE
================================================================================
Priority: TIER 3 — Proposal alignment, nice-to-have
Where:    free_space.py (FreeSpaceGenerator)
Depends:  None
Effort:   2-3 days

What:  The proposal (Section 4.2) says: "If new objects are found
       within a partition, the process repeats: bound the biggest
       object and define its occluded space."

       The actual FreeSpaceGenerator does a single-pass octree that
       marks cells as FREE or OCCUPIED.  It never detects new objects
       inside partitions — all objects are detected upfront by
       oracle_detect_objects.

       For the current tabletop scenario with a fixed overhead camera,
       all objects ARE detected upfront, so this gap is not triggered.
       It would matter for partial-coverage sensors or objects hidden
       inside containers.

Fix:   ONE COMMIT — add recursive discovery pass:
       (a) After initial octree subdivision, for each OCCUPIED cell,
           check if it contains objects not yet in the registry.
       (b) If new objects found, create object/shadow boxels for them
           and repeat subdivision.
       (c) Terminate when no new objects are discovered.

       Alternatively, document as an accepted simplification for the
       fixed-camera scenario and note as future work in the thesis.

Related: #4 (re-boxelization could trigger re-discovery)
