CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET GUI FREEZES DURING PLANNING — THIS IS NORMAL
       The RenderingLock (robot_utils.py) disables COV_ENABLE_RENDERING
       for the entire planning phase (test_full_pipeline.py ~line 346,
       audit #60).  While planning runs (often 30-130 s), the PyBullet
       window appears frozen — no zoom, pan, or rotation works.
       The GUI unfreezes when planning finishes.
       If planning takes very long, WSLg's Weston compositor may drop
       the X11 connection, crashing the process with:
         XIO: fatal IO error 11 (Resource temporarily unavailable) on X server ":0"
       Fix: restart WSL (wsl --shutdown from PowerShell, then reopen
       terminal).  Keyboard camera controls (R/F for zoom, WASD for
       pan) also do not work while rendering is locked.

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

STATIC GEOMETRY IN INIT (e.g. boxel_fits) — NOW REBUILT EACH REPLAN
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot (old #102).  They are valid only for that partition and
       geometry for one planning problem.

       As of 2026-04-15, free-space boxels ARE re-merged and re-split
       between replans (reboxelize_free_space after sense).  The registry
       is mutated in place before each planner.plan() call, so
       _build_init() always reads the latest partition.  Shadow removal
       and new object/shadow creation after sense also update the
       registry before the next replan.  The planner is stateless
       between calls — all context is passed in each plan() call.

BOXEL_FITS (old #102) — STATIC INIT, NOT A TEST STREAM
       The place precondition (boxel_fits ?o ?b) is satisfied by init facts
       only.  BoxelStreams.test_boxel_fits() runs once per problem in
       _build_init() for each (object, free_space) pair that geometrically
       fits.  An earlier implementation used a PDDLStream test stream
       (test-boxel-fits); adaptive search re-evaluated it heavily across
       skeletons and dominated runtime despite a cheap per-call check.
       stream.pddl no longer declares that stream; domain and predicate are
       unchanged.

#13. RECURSIVE OBJECT DISCOVERY IN FREE SPACE
================================================================================
Priority: TIER 3 — Proposal alignment, nice-to-have
Where:    free_space.py (FreeSpaceGenerator)
Depends:  None
Effort:   2-3 days

What:  The proposal (Section 4.2) says: "If new objects are found
       within a partition, the process repeats: bound the biggest
       object and define its occluded space."

       The actual FreeSpaceGenerator does a single-pass octree that
       marks cells as FREE or OCCUPIED.  It never detects new objects
       inside partitions — all objects are detected upfront by
       oracle_detect_objects.

       For the current tabletop scenario with a fixed overhead camera,
       all objects ARE detected upfront, so this gap is not triggered.
       It would matter for partial-coverage sensors or objects hidden
       inside containers.

Fix:   ONE COMMIT — add recursive discovery pass:
       (a) After initial octree subdivision, for each OCCUPIED cell,
           check if it contains objects not yet in the registry.
       (b) If new objects found, create object/shadow boxels for them
           and repeat subdivision.
       (c) Terminate when no new objects are discovered.

       Alternatively, document as an accepted simplification for the
       fixed-camera scenario and note as future work in the thesis.

Related: #4 (re-boxelization could trigger re-discovery)

#14. SPARSE VISIBILITY CHECK — 8 AABB CORNERS MISS PARTIALLY VISIBLE OBJECTS
================================================================================
Priority: TIER 0 — Root cause of persistent sensing failures
Where:    boxel_env.py (oracle_detect_objects, ~line 900)
Depends:  None
Effort:   1-2 days

What:  oracle_detect_objects() determines whether an object is visible
       by casting exactly 8 rays from the camera to the 8 corners of
       the object's AABB.  If ANY ray reaches the object, it is
       visible; if ALL 8 are intercepted, the object is classified as
       invisible.

       8 corners is too sparse.  When an object is partially visible
       (e.g. a sliver sticks out between two occluders, or the top
       protrudes above a shorter occluder), none of the 8 corners may
       land in the visible gap.  The object is then classified as
       invisible despite being partially visible from the camera.

       Observed failure (run 2026-04-03): occluder_1 at (0.5, 0.2)
       was classified as invisible because all 8 AABB corner rays
       were intercepted by occluder_2 and occluder_3 in front.
       Consequences:
       (a) occluder_1 never entered the BoxelRegistry as an object
           boxel — the planner had no knowledge of it.
       (b) Shadow volumes from other occluders extended through
           occluder_1's position unsplit (#15).
       (c) The planner could never plan to relocate occluder_1,
           causing ALL subsequent sense actions to report 49-58%
           ray blockage.  10 plans executed, all failed — total
           mission failure.

Fix:   ONE COMMIT — increase ray density in oracle_detect_objects():
       (a) Replace 8-corner rays with a denser grid over the AABB
           surface (e.g. 4×4 grid per face = 96 rays, or N×N on the
           camera-facing face only).
       (b) Alternative: use the rendered depth buffer to check if
           any pixel corresponds to the object's body ID (segmentation
           mask approach — PyBullet's getCameraImage returns a
           segmentation buffer).  This is both cheaper and catches
           any visible pixel, no matter how small.
       (c) Either approach should use a visibility THRESHOLD: an
           object may be "visible" if ≥ K% of sample points reach
           it, to filter noise from sub-pixel slivers.

Related: #15 (shadow splitting depends on correct visibility),
         #16 (camera angle exacerbates this)

#21. BLOCKED SENSE CAUSES SHADOW SWITCHING AND FALSE "SEARCHED ALL" TERMINATION
================================================================================
Priority: TIER 0 — Control-loop correctness bug
Where:    test_full_pipeline.py (BeliefState, replanning loop, still_blocked handling)
Depends:  None
Effort:   1-2 days

What:  After a pick/place intended to clear a shadow, sensing can still return
       "still_blocked".  The execution loop then breaks to replan, but it does
       NOT preserve an obligation to keep clearing that SAME shadow.

       Current behavior:
       (a) belief.occluders_moved records the relocated occluder.
       (b) The blocked shadow remains unknown unless blocked_counts reaches 3.
       (c) The next plan is free to target a DIFFERENT unknown shadow instead
           of identifying and relocating the blocker that still obstructs the
           current one.
       (d) After 3 consecutive "still_blocked" outcomes on the same shadow,
           the loop marks that shadow as not_here even though the view was
           never clear and the shadow was never actually observed.
       (e) The run can then terminate with "Searched all shadows but target not
           found" even though some shadows were only blocked, not searched.

       This is different from #5.  #5 is about choosing a placement
       destination that re-blocks a corridor.  This issue is about the
       REACTIVE POLICY after sensing reports that a corridor is still blocked:
       the system switches search targets instead of continuing to clear the
       active shadow.

Fix:   ONE COMMIT — design decision needed:

       OPTION A — Introduce a belief/planning obligation to continue clearing
       the same shadow after still_blocked, until view_clear or clear_but_empty.
       OPTION B — Add a distinct shadow status such as blocked_unresolved;
       do NOT map it to not_here, and prioritize it before untouched shadows.
       OPTION C — On still_blocked, identify the current blocking object(s)
       for that same shadow and force replanning around one of those blockers.

       OPTION B + C is likely the most practical: preserve truthful belief
       state and bias replanning toward the active blocked shadow.

       DESIGN DIRECTION (2026-04-17): The fix MUST be in the PDDL model,
       not in the execution loop.  The planner should reason about blocked
       shadows.  Specifically:
       - Add a blocked_unresolved shadow status (distinct from unknown
         and not_here) in both BeliefState and the PDDL model.
       - The PDDL domain must represent that a shadow is blocked and
         needs further occluder removal before sensing can succeed.
       - The planner should be responsible for selecting which shadow to
         clear next — not the reactive execution loop.
       - The execution loop must NEVER mark an unobserved shadow as
         not_here under any circumstance (remove the 3-strike give-up).

Related: #5 (placement re-blocks cleared corridor), #4 (stale world model),
         archive resolved #78(c) (3-strike give-up behavior)

#30. GOAL AS CLI PARAMETER — STACK GOAL ADDED, PERF DECISION OPEN [PARTIAL 2026-04-24]
================================================================================
Priority: TIER 3 — Evaluation infrastructure / extensibility
Where:    test_full_pipeline.py (main, CLI), pddl/domain_pddlstream.pddl,
          pddl/stream.pddl, streams.py, pddlstream_planner.py,
          execution.py, boxel_env.py
Depends:  None
Effort:   IMPLEMENTED 2026-04-24 (commit 0d5def7); keep/kill decision
          pending mitigation experiment (split pick + unstack)

What:  The goal was hardcoded as ('holding', target_name).  It should
       be passable as a CLI parameter (like --show-free), with stacking
       (('on', obj_a, obj_b) and conjunctions) as a first additional
       goal kind.

DONE 2026-04-24 (commit 0d5def7):
       Added --goal {holding,stack}, --stack-height (>=2), --n-objects.
       Domain: new (on ?o1 ?o2), (clear ?o), (stack_kin ?o ?on ?g ?q)
       predicates; new stack action; conditional pick effect re-clears
       the support; :conditional-effects requirement added.
       Stream: compute-stack-kin reads ?on_obj's live AABB so multi-step
       stacks tolerate per-step settling.
       Planner: _build_init emits (clear/on) ONLY when stackable_objects
       is supplied.  Holding-goal callers leave it None and pay zero
       extra grounding cost on the init side.
       Execution: execute_stack mirrors execute_place but resolves
       destination from live support AABB (60-step settle vs 30).
       Scene: stack_scene + constrain_to_reach intersects safe on-table
       window (_SAFE_TABLE_*_RANGE = (0.05..0.70, -0.40..0.40)) with
       Panda reach disk so cubes never spawn off-table.

       Smoke tests:
         --goal stack --stack-height 2 --no-gui --seed 0  → 1 plan,
           1.37 s, plan = move/pick/move/stack ✓
         --goal stack --stack-height 3 --no-gui --seed 0  → 1 plan,
           1.58 s, plan = move/pick/move/stack/move/pick/move/stack ✓
           (collision warnings during 2nd pick are runtime artifacts of
           the conditional pick effect not telling execution that
           red_object will fall once orange_object is picked from under
           it; goal is still achieved correctly)
         --goal stack --stack-height 2 --gui --seed 0  → 1 plan,
           1.17 s, identical plan ✓

PERFORMANCE PROBLEM (audit #30 keep/kill measurement):
       Even though init facts are gated, the holding-goal per-call
       planning time roughly DOUBLES on the same scene/seed:

         Baseline (commit fed063a, pre-stack), default scene, no-GUI:
           seed 0  → 5 plans, 20.97s,  4.19 s/plan
           seed 1  → 7 plans, 37.77s,  5.40 s/plan
           seed 2  → 1 plan,   4.68s,  4.68 s/plan
           seed 3  → 2 plans,  6.06s,  3.03 s/plan
           Aggregate (15 calls): 4.63 s/call
         GUI variant (seed 2):  13 plans, 58.98s, 4.54 s/plan
           → confirms RenderingLock isolates plan() from GUI overhead.

         After stack feature (commit 0d5def7), default scene, holding:
           no-GUI seed 0 → 3 plans, 31.83s, 10.61 s/plan  (+153%)
           GUI    seed 0 → 2 plans, 15.76s,  7.88 s/plan  (+88%)
           GUI    seed 0 → 1 plan,   8.36s,  8.36 s/plan  (+99%)
           Three independent seed-0 runs: 7.9–10.6 s/call vs 4.2 s/call.
           Consistent ~2x per-call slowdown.  Way past the 10% budget
           the user accepted before starting.

       Plan COUNTS differ across runs because stack code introduces
       extra random.shuffle paths in scene RNG (different targets get
       picked); cumulative time isn't directly comparable, but the
       per-call number is the headline regression metric.

       Run-by-run timing summaries live in
         logs/run_<timestamp>/timing_summary.json
       (see commit fed063a for the instrumentation).

LIKELY ROOT CAUSE:
       The conditional pick effect

         (forall (?x)
           (when (on ?o ?x)
             (and (not (on ?o ?x)) (clear ?x))))

       grounds against every (Obj ?x) for every pick instantiation,
       regardless of whether (on ?o ?x) is ever asserted.  Combined
       with the :conditional-effects requirement (which switches
       FastDownward to a slower translator path), this is the most
       likely source of the ~2x per-call slowdown.  Init-side gating
       does not help because the cost is in the domain grounder.

PROPOSED MITIGATION (NOT YET TRIED — needs user approval):
       Split pick into pick + unstack:
         - Remove the forall from pick.
         - Drop :conditional-effects from the requirements list.
         - Add a separate (:action unstack) with precondition
           (on ?o ?support) and effect that adds (clear ?support).
         - Both pick and unstack map to the same execute_pick routine
           in the dispatcher (symbolically different, physically same).
       Holding-goal runs would then never see (on ?o ?x), the
       grounder would discard unstack early, and the domain shape for
       holding goals would be effectively identical to pre-stack.

KEEP / KILL OPEN QUESTIONS:
       - Will the split-pick mitigation recover holding-goal perf to
         within 10% of the pre-stack baseline?  Needs measurement.
       - If not, do we accept ~2x per-call slowdown for the sake of
         supporting stacking goals in evaluation runs (audit #9)?
       - Or revert commit 0d5def7 and revisit later when we have a
         contingent planner / different domain encoding that avoids
         the conditional-effects cost?

Related: #9 (evaluation needs both 'holding' and 'stack' goal kinds for
         baseline comparisons), #36 (holding goal terminates at contact
         — stack goal is a natural extension that already implies an
         upright lift via the next move).

#33. REMOVE_BOXEL_VIZ USES O(N) LIST.REMOVE()  [DONE 2026-04-24]
================================================================================
Priority: TIER 3 — Performance (visualization)
Where:    visualization.py (BoxelVisualizer.remove_boxel_viz, clear_all)
Depends:  None
Effort:   <1 day

What:  remove_boxel_viz() called self.debug_items.remove(item_id) and
       self.shadow_bodies.remove(body_id), both O(n) per call.  With
       many boxels, incremental removal accumulated quadratic cost

#37. EXECUTE_PICK STILL RE-SOLVES IK INSTEAD OF USING PLANNER'S q
================================================================================
Priority: TIER 3 — Very low.  Long debate, no agreed conclusion.
Where:    streams.py (compute_kin_solution, _GRASP_Z_OFFSETS),
          execution.py (execute_pick, execute_place),
          test_full_pipeline.py (pick/place dispatch handlers),
          pddl/stream.pddl, pddl/domain_pddlstream.pddl
Depends:  None (orthogonal to all other open issues)
Effort:   Unknown — depends on which design wins.

WARNING TO FUTURE AI SESSIONS:
       This was discussed at length on 2026-04-22.  Many proposed
       refactors were rejected after design review.  Re-read this entry
       BEFORE proposing yet another redesign.  Specifically, the
       following ideas were considered AND REJECTED (see "Rejected
       directions" below); do not reintroduce them without new
       evidence:
         - Boxelizing the target after found_target sense outcome.
         - A `get-object-pose(?o)` PDDLStream stream that yields live
           env.objects poses.
         - An `observed_objects: Set[str]` gate inside BoxelStreams.
         - Restructuring pick into (?o ?b ?g ?p ?q1 ?q2 ?t) to mirror
           the move action.

What:  compute_kin_solution returns a config q solved against
       boxel.center + grasp.position (with grasp.position[2] = 0.10 m,
       i.e. the APPROACH pose, 10 cm above the boxel center).  The
       planned `move` action delivers the arm to q.  execute_pick then
       IGNORES q and locally re-solves IK for the actual contact pose
       using env.objects[obj_name].position.  The same applies to
       execute_place.  This duplicates work already done by the
       planner and bypasses plan_motion's collision checks for the
       lowering segment.

       The 2026-04-22 series of refactor steps (1, 2 — completed)
       removed the post-grasp lift and the redundant pre-contact
       approach motion.  Step 5 was meant to remove the local IK
       lowering as well, by changing _GRASP_Z_OFFSETS so that q
       targets the contact point directly.  This step was DEFERRED
       because of the gripper-state-during-move concern (see below).

Rejected directions (do not re-propose):

  (1) Boxelize the discovered target on found_target.
      Asymmetry between `contains_nontarget` (creates OBJECT + SHADOW
      boxels for newly-discovered non-targets) and `found_target`
      (no registry change, just belief.mark_sensed) is INTENTIONAL.
      Non-targets become future obstacles the planner must reason
      about; targets are about to be picked and removed.  Creating a
      registry entry just to delete it one action later is wasteful
      and would force an extra replan cycle.  The pick handler's
      `elif obj_str == target_name:` branch (test_full_pipeline.py
      ~line 797) reads env.objects[target].position directly — this
      is the ACCEPTED design.

  (2) New `get-object-pose(?o)` stream that yields live env.objects
      poses for use in compute-kin.  Rejected because compute-kin
      already uses boxel.center, which is the correct abstraction
      under the "trust boxel data freshness" assumption (see open
      sub-issue below).  Reading env.objects from a stream would be
      planning-time cheating: the planner would have access to
      ground-truth positions of objects independent of what it has
      observed.

  (3) `observed_objects: Set[str]` partial-observability gate inside
      BoxelStreams.  Unnecessary because registry membership already
      gates compute-kin: hidden objects without OBJECT boxels cannot
      be IK targets, so the gate is implicit in the data structure.

  (4) Restructure pick to (?o ?b ?g ?p ?q1 ?q2 ?t) mirroring move.
      Discussed but not pursued — too large a PDDL surgery for
      the value gained, and the cleaner formulation depends on
      decisions about place semantics (single compute-kin variant
      vs. two, boxel-as-pose coercion) that were not resolved.

Tentatively agreed but NOT implemented (option 5a):

  (a) Set _GRASP_Z_OFFSETS = [0.02] (was [0.10]).  q now targets a
      point 2 cm above the object — close enough that the lowering
      segment can be a fixed linear interpolation rather than a
      separate IK call.
  (b) Replace the local solve_ik() call in execute_pick with a
      simple 2 cm joint-space lower from the current pose, using
      cached IK seeds or Cartesian linear interpolation.
  (c) Same change to execute_place.
  (d) Open question: gripper state during the planned move.  With
      offset 0 the arm arrives at contact with whatever gripper
      state it had — if closed, it would push the object during
      the trajectory; if opened at contact, fingers spread inside
      the object.  Two unresolved options:
        (i) Keep small clearance (option 5a above, 2 cm) so opening
            at the end of move is safe.
        (ii) Pre-emptively open the gripper in the dispatcher when
             the next planned action is `pick`, so the arm arrives
             at contact already open.

Sub-issue (related, also unresolved): boxel data staleness.
       compute_kin_solution assumes boxel.center is current.  After
       an object settles in physics, drifts, or is moved by a
       previous plan iteration, the OBJECT boxel's center may be
       stale until the next reboxelization.  Currently we accept this
       drift; if it causes pick failures, add a periodic re-AABB
       sync between actions (cheap p.getAABB() per OBJECT boxel,
       no full reboxelization).  Not tracked as a separate audit
       item — bundled here.

Why this is low priority:  the current code works.  The local IK
       in execute_pick gives correct contact poses, and the
       constraint-based grasp absorbs small EE/object misalignment
       (audit #1 / #98).  The refactor is an aesthetics + 1:1
       PDDL-execution-mapping concern, not a correctness one.  Visit
       only when there is a concrete reason (e.g. planning with a
       different motion planner, or extending to a goal that requires
       multiple consecutive pick/place chains).

Related: execution.py refactor 2026-04-22 steps 1 & 2,
         audit #1 (RESOLVED, archive — contact-height fix),
         audit #36 (holding goal terminates at contact)

#39. PICK DOES NOT REQUIRE (CLEAR ?O) — STACK-BASE PICK TOPPLES TOWER SILENTLY

Where:    pddl/domain_pddlstream.pddl (:action pick ~line 188),
          pddlstream_planner.py (_build_init — clear init for
          stackable_objects, ~line 490-495)
Depends:  None (but fix interacts with #30 "split pick + unstack"
          mitigation — coordinate before committing)
Effort:   ~0.5 day (precondition addition + holding-goal gating)

What:  The pick action's preconditions do not include (clear ?o).
       When a tower A-on-B-on-C-on-table exists and pick is invoked
       on B or C, all preconditions remain satisfied: (obj_at_boxel
       B B) is still true (the tower doesn't symbolically move B's
       boxel); (handempty), (at_config ?q), and (kin_solution ...)
       are independent of whether something is stacked on B.  The
       forall (when (on ?o ?x) (and (not (on ?o ?x)) (clear ?x)))

       objects resting ON ?o.

       --stack-height 5, seed 0):

                                    Physical: orange+red tower

                                    then.  Same bug, again.

         the subsequent move and stack steps (the robot drags
         through the fallen cubes), yet the run terminates with
         SUCCESS because the final SYMBOLIC state matches the goal
         (see #40 — the success check is symbolic, not physical).
         Physically 3 of 4 (on ...) goal clauses were violated.

Fix:   ONE COMMIT:
       (a) Add (clear ?o) to pick's precondition in
           pddl/domain_pddlstream.pddl (:action pick).
       (b) Emit (clear X) in init for every Obj unconditionally, not
           only when stackable_objects is supplied

           less than the conditional-effects cost already debated
           in #30.

             (or (not (stack_mode_enabled)) (clear ?o))

           effect that already assumes (clear ?x) is well-defined.
       (c) Re-run --goal stack --stack-height 5 --seed 0 and verify
           the planner builds bottom-up (green base, blue on green,
           cyan on blue, orange on cyan, red on orange) instead of
           the observed stack-and-disassemble pattern.
       (d) Coordinate with #30 mitigation: if "split pick + unstack"
           is eventually pursued, unstack KEEPS (clear ?o) false
           and (on ?o ?support) as its precondition; plain pick gets
           (clear ?o).  Either fix should be consistent with that.

#40. FALSE-SUCCESS IN STACK GOALS — SYMBOLIC STATE ACCEPTED OVER PHYSICS
================================================================================
Priority: TIER 0 — Correctness bug (reports SUCCESS on physically failed runs)
Where:    test_full_pipeline.py (post-execution goal check, ~line 1009+;
          on_relations tracking in the stack dispatch handler)
Depends:  None.  Independent of #39 — even if pick is fixed, a cube can
          slip off during subsequent physics settling and produce the
          same desync.
Effort:   ~0.5-1 day

What:  After plan execution terminates, the pipeline compares an
       on_relations dict (built by the executor from what each stack
       action SAID happened) to the goal conjunction.  The dict is
       never reconciled with physics.  Any PDDL-level symbolic
       success is printed as SUCCESS regardless of physical state.

       Failure observed (same run as #39):

         Final on-relations logged by pipeline:
           orange_object → cyan_object    (physically on floor)
           red_object    → orange_object  (physically on floor)
           cyan_object   → blue_object    (physically on floor)
           blue_object   → green_object   (actually true)
         Pipeline prints: SUCCESS!

       Consequences:
       (a) Evaluation runs (audit #9) would inflate success rates.
       (b) Debugging is harder — the log says SUCCESS, so manual
           GUI review is needed to know what really happened.
       (c) Any PDDL-vs-physics desync (#39, grasp slip, settling
           drift, adversarial scene state) produces false-success.

Fix:   ONE COMMIT — add a physics-based goal verifier:
       (a) After symbolic success, walk the goal AST.  For each
           clause:
             (on A B): query p.getAABB for both A and B; require
               A's bottom_z ≈ B's top_z (ε ~5 mm) AND A's (x,y)
               centre within B's (x,y) AABB.
             (holding X): check the grasp constraint still exists
               and targets X (body id match).
             (at_config q): check current joints ≈ q (RRT
               tolerance).
             (on_table X)  (if added in #41): check X's bottom_z
               ≈ table top_z.
       (b) On mismatch, print "PHYSICAL_FAILURE: <clause> —
           symbolic true but physics disagrees (<details>)" and
           set the run exit status to failure.
       (c) Log mismatch details in timing_summary.json so
           evaluation tooling (#9) can filter false-positives.
       (d) Choose tolerance ε carefully: too tight → physics-true
           runs flagged; 5 mm on the AABB touch test is a
           reasonable default for 40-mm cubes.

       Optional: also run the verifier after EACH stack/place
       action, break to replan on mismatch.  This would MASK #39
       by replanning around the collapse — useful fallback but
       does not remove the need for #39's precondition fix.

Related: #39 (root cause of this run's false-success),
         #9  (evaluation metrics depend on truthful success flags),
         #21 (reactive policy after unexpected state — similar
              loop-vs-planner design question),
         #41 (on_table clause would also need a verifier case).

#41. STACK GOAL DOES NOT CONSTRAIN BASE TO BE ON THE TABLE
================================================================================
Priority: TIER 3 — Minor, related to #40
Where:    test_full_pipeline.py (build_stack_goal ~line 111),
          pddl/domain_pddlstream.pddl (new predicate or convention)
Depends:  None.  Could ride with #40's verifier work.
Effort:   <1 day

What:  build_stack_goal produces goals of the form
       (and (on t_1 t_2) (on t_2 t_3) ... (on t_{H-1} t_H))
       with no clause asserting that t_H (the base) is on the
       table.  Currently this is moot: all cubes start on the
       table; no action lifts a cube off the table without
       replacing it somewhere; so the base is always table-resting
       by induction.

       However it is still a correctness gap:
       (a) If #39 is ever partially fixed but not consistently,
           pick/place sequences could leave the base floating
           (ungrounded stack).
       (b) A future plan shape (e.g. pick the base cube, sense
           under it, then re-stack) would let the base end up
           anywhere.
       (c) The goal is semantically incomplete: a reader cannot
           tell from the goal whether "a tower" is intended or
           just a chain of (on) relations in mid-air.

Fix:   ONE COMMIT — options:
       (a) Add an (on_table ?o) predicate to the domain, emit
           (on_table X) in init for every cube, and have
           pick/place/stack effects maintain it (pick removes,
           stack removes, place emits).  build_stack_goal adds
           (on_table t_H) to the conjunction.  Verifier (#40)
           extends to this clause.
       (b) Represent the table as an Obj (with is_table flag)
           and use (on t_H table) with a is_table constant.  More
           symmetric; bigger domain change.
       Option (a) is lighter-touch and compatible with the
       existing domain.

Related: #40 (physics verifier should check the on_table clause),
         #30 (stack feature that needs this completion),
         pddl/domain_pddlstream.pddl comment at ~line 64:
           "Objects not mentioned in any (on ?o ?x) fact is treated
            as table-resting" — this convention becomes explicit.

#42. EXECUTION COLLISIONS ARE LOGGED BUT DO NOT TRIGGER REPLAN
================================================================================
Priority: TIER 3 — Robustness
Where:    execution.py (collision detection around each action),
          test_full_pipeline.py (execution loop break-to-replan logic)
Depends:  None (but #4 re-boxelization work affects what a post-
          collision replan would see)
Effort:   ~1 day

What:  When execute_pick / execute_place / execute_stack / move
       detect robot-vs-object collisions, they print
       "*** COLLISION [<context>]: robot link X <-> Y ***" and
       continue execution.  Nothing in the execution loop
       propagates the collision to a break-to-replan.

       Observed in run 2026-04-26_11-11-08 (stack-height=5 run
       that triggered #39/#40): 13+ collision warnings across
       "after pick cyan_object", "move to blue_object", etc.,
       as the robot ploughed through the debris from the #39
       tower collapse.  Execution continued to the end and
       reported SUCCESS via the #40 false-success path.

       Even if #39 and #40 are fixed, collisions can arise from:
       (a) settling drift (small; usually no damage)
       (b) adversarial boxel/AABB inaccuracy after an occluder
           relocation (#4 shadow recompute gap)
       (c) multi-object dropping when a pick-under-tower bug
           exists in any future action.

Fix:   ONE COMMIT:
       (a) Count collisions per action in execute_* routines;
           return a collision_count alongside the return config.
       (b) Execution loop: if collision_count > threshold (e.g.
           >0 for pick/place/stack, >N for move), break to replan
           and update BeliefState with the observed scene state
           (object positions after settling).
       (c) Make the threshold configurable via CLI (e.g.
           --strict-collisions for thesis evaluation, relaxed
           default for dev).

Related: #4 (re-boxelization after unexpected state),
         #21 (reactive policy for unexpected outcomes — same
              loop-vs-planner design question),
         #39/#40 (the toppling-tower class of failure this
                  would catch even when #39 is fixed but a
                  different physics bug emerges).
