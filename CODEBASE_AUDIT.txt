CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

#13. RECURSIVE OBJECT DISCOVERY IN FREE SPACE
================================================================================
Priority: TIER 3 — Proposal alignment, nice-to-have
Where:    free_space.py (FreeSpaceGenerator)
Depends:  None
Effort:   2-3 days

What:  The proposal (Section 4.2) says: "If new objects are found
       within a partition, the process repeats: bound the biggest
       object and define its occluded space."

       The actual FreeSpaceGenerator does a single-pass octree that
       marks cells as FREE or OCCUPIED.  It never detects new objects
       inside partitions — all objects are detected upfront by
       oracle_detect_objects.

       For the current tabletop scenario with a fixed overhead camera,
       all objects ARE detected upfront, so this gap is not triggered.
       It would matter for partial-coverage sensors or objects hidden
       inside containers.

Fix:   ONE COMMIT — add recursive discovery pass:
       (a) After initial octree subdivision, for each OCCUPIED cell,
           check if it contains objects not yet in the registry.
       (b) If new objects found, create object/shadow boxels for them
           and repeat subdivision.
       (c) Terminate when no new objects are discovered.

       Alternatively, document as an accepted simplification for the
       fixed-camera scenario and note as future work in the thesis.

Related: archived #4 (re-boxelization — recursive pass optional if
         partial-coverage perception is added).

#21. BLOCKED SENSE CAUSES SHADOW SWITCHING AND FALSE "SEARCHED ALL" TERMINATION
================================================================================
Priority: TIER 0 — Control-loop correctness bug
Where:    test_full_pipeline.py (BeliefState, replanning loop, still_blocked handling)
Depends:  None
Effort:   1-2 days

What:  After a pick/place intended to clear a shadow, sensing can still return
       "still_blocked".  The execution loop then breaks to replan, but it does
       NOT preserve an obligation to keep clearing that SAME shadow.

       Current behavior:
       (a) belief.occluders_moved records the relocated occluder.
       (b) The blocked shadow remains unknown unless blocked_counts reaches 3.
       (c) The next plan is free to target a DIFFERENT unknown shadow instead
           of identifying and relocating the blocker that still obstructs the
           current one.
       (d) After 3 consecutive "still_blocked" outcomes on the same shadow,
           the loop marks that shadow as not_here even though the view was
           never clear and the shadow was never actually observed.
       (e) The run can then terminate with "Searched all shadows but target not
           found" even though some shadows were only blocked, not searched.

       This is different from #5.  #5 is about choosing a placement
       destination that re-blocks a corridor.  This issue is about the
       REACTIVE POLICY after sensing reports that a corridor is still blocked:
       the system switches search targets instead of continuing to clear the
       active shadow.

Fix:   ONE COMMIT — design decision needed:

       OPTION A — Introduce a belief/planning obligation to continue clearing
       the same shadow after still_blocked, until view_clear or clear_but_empty.
       OPTION B — Add a distinct shadow status such as blocked_unresolved;
       do NOT map it to not_here, and prioritize it before untouched shadows.
       OPTION C — On still_blocked, identify the current blocking object(s)
       for that same shadow and force replanning around one of those blockers.

       OPTION B + C is likely the most practical: preserve truthful belief
       state and bias replanning toward the active blocked shadow.

       UPDATE (2026-05-02, see #45): the contains_nontarget variant of
       the same observable failure (sense reveals an object that turns
       out to be the shadow's own occluder) is now blocked at the
       blocker-detection layer — compute_shadow_blockers no longer
       reports view_clear for those shadows, so the planner picks the
       occluder before sensing.  The still_blocked control-loop bug
       described below is not subsumed by that fix and remains open.

       DESIGN DIRECTION (2026-04-17): The fix MUST be in the PDDL model,
       not in the execution loop.  The planner should reason about blocked
       shadows.  Specifically:
       - Add a blocked_unresolved shadow status (distinct from unknown
         and not_here) in both BeliefState and the PDDL model.
       - The PDDL domain must represent that a shadow is blocked and
         needs further occluder removal before sensing can succeed.
       - The planner should be responsible for selecting which shadow to
         clear next — not the reactive execution loop.
       - The execution loop must NEVER mark an unobserved shadow as
         not_here under any circumstance (remove the 3-strike give-up).

Related: #5 (placement re-blocks cleared corridor),
         archive resolved #78(c) (3-strike give-up behavior)

LIKELY ROOT CAUSE:
       The conditional pick effect

         (forall (?x)
           (when (on ?o ?x)
             (and (not (on ?o ?x)) (clear ?x))))

       grounds against every (Obj ?x) per pick instantiation, and the
       :conditional-effects requirement switches FastDownward to a
       slower translator path.  Init-side gating does not help — cost
       is in the domain grounder.

PROPOSED MITIGATION (NOT YET TRIED — needs user approval):
       Split pick into pick + unstack:
         - Remove the forall from pick; drop :conditional-effects.
         - Add (:action unstack) with precondition (on ?o ?support)
           that emits (clear ?support) as effect.
         - Both map to the same execute_pick routine.
       Holding-goal runs would then never see (on ?o ?x); the grounder
       would discard unstack early.  Forward-compatible with #39 (DONE
       2026-04-26): plain pick already gets (clear ?o) either way.

KEEP / KILL OPEN:
       - Does split-pick recover holding-goal perf to within 10%?
         (needs measurement)
       - If not, accept ~2x slowdown for stacking support (audit #9)?
       - Or revert commit 0d5def7 and revisit with a different domain
         encoding that avoids the conditional-effects cost?

Related: #9 (evaluation needs both 'holding' and 'stack' goal kinds),

#40. FALSE-SUCCESS IN STACK GOALS — SYMBOLIC STATE ACCEPTED OVER PHYSICS
================================================================================
Priority: TIER 0 — Correctness bug (reports SUCCESS on physically failed runs)
Where:    test_full_pipeline.py (post-execution goal check, ~line 1009+;
          on_relations tracking in the stack dispatch handler)
Depends:  None.  Independent of #39 — even if pick is fixed, a cube can
          slip off during subsequent physics settling and produce the
          same desync.
Effort:   ~0.5-1 day

What:  After plan execution terminates, the pipeline compares an
       on_relations dict (built by the executor from what each stack
       action SAID happened) to the goal conjunction.  The dict is
       never reconciled with physics.  Any PDDL-level symbolic
       success is printed as SUCCESS regardless of physical state.

       Failure observed (same run as #39):

         Final on-relations logged by pipeline:
           orange_object → cyan_object    (physically on floor)
           red_object    → orange_object  (physically on floor)
           cyan_object   → blue_object    (physically on floor)
           blue_object   → green_object   (actually true)
         Pipeline prints: SUCCESS!

       Consequences:
       (a) Evaluation runs (audit #9) would inflate success rates.
       (b) Debugging is harder — the log says SUCCESS, so manual
           GUI review is needed to know what really happened.
       (c) Any PDDL-vs-physics desync (#39, grasp slip, settling
           drift, adversarial scene state) produces false-success.

Fix:   ONE COMMIT — add a physics-based goal verifier:
       (a) After symbolic success, walk the goal AST.  For each
           clause:
             (on A B): query p.getAABB for both A and B; require
               A's bottom_z ≈ B's top_z (ε ~5 mm) AND A's (x,y)
               centre within B's (x,y) AABB.
             (holding X): check the grasp constraint still exists
               and targets X (body id match).
             (at_config q): check current joints ≈ q (RRT
               tolerance).
             (on_table X)  (once #41 adds this atom to goals/init): check
               X's bottom_z ≈ table top_z.
       (b) On mismatch, print "PHYSICAL_FAILURE: <clause> —
           symbolic true but physics disagrees (<details>)" and
           set the run exit status to failure.
       (c) Log mismatch details in timing_summary.json so
           evaluation tooling (#9) can filter false-positives.
       (d) Choose tolerance ε carefully: too tight → physics-true
           runs flagged; 5 mm on the AABB touch test is a
           reasonable default for 40-mm cubes.

       Optional: also run the verifier after EACH stack/place
       action, break to replan on mismatch.  This would MASK #39
       by replanning around the collapse — useful fallback but
       does not remove the need for #39's precondition fix.

Related: #39 (root cause of this run's false-success),
         #9  (evaluation metrics depend on truthful success flags),
         #21 (reactive policy after unexpected state — similar
              loop-vs-planner design question),
         #41 (on_table clause in goal — verifier case above).

          pddl/domain_pddlstream.pddl (new predicate or convention)

What:  build_stack_goal produces goals of the form
       (and (on t_1 t_2) (on t_2 t_3) ... (on t_{H-1} t_H))
       with no clause asserting that t_H (the base) is on the
       table.  Currently this is moot: all cubes start on the
       table; no action lifts a cube off the table without
       replacing it somewhere; so the base is always table-resting
       by induction.

       However it is still a correctness gap:
       (a) If #39 is ever partially fixed but not consistently,
           pick/place sequences could leave the base floating
           (ungrounded stack).
       (b) A future plan shape (e.g. pick the base cube, sense
           under it, then re-stack) would let the base end up
           anywhere.
       (c) The goal is semantically incomplete: a reader cannot
           tell from the goal whether "a tower" is intended or
           just a chain of (on) relations in mid-air.

Fix:   ONE COMMIT — options:
       (a) Add an (on_table ?o) predicate to the domain, emit
           (on_table X) in init for every cube, and have
           pick/place/stack effects maintain it (pick removes,
           stack removes, place emits).  build_stack_goal adds
           (on_table t_H) to the conjunction.  Verifier (#40)
           extends to this clause.
       (b) Represent the table as an Obj (with is_table flag)
           and use (on t_H table) with a is_table constant.  More
           symmetric; bigger domain change.
       Option (a) is lighter-touch and compatible with the
       existing domain.

Related: #40 (physics verifier should check the on_table clause),
         #30 (stack feature that needs this completion),
         pddl/domain_pddlstream.pddl comment at ~line 64:
           "Objects not mentioned in any (on ?o ?x) fact is treated
            as table-resting" — this convention becomes explicit.
