CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

META-ISSUE — OLD #95 SYSTEMIC PDDL STATE DRIFT (dissolved)
       The 4 drift channels between PDDL symbolic state and PyBullet

- Parallel FastDownward / GPU robotics sim — post-thesis research
  direction.  Idea raised 2026-05-10 (later) by user; web-searched
  same day for prior work, findings below.  Bullet refined from
  "speculative" to "mostly-solved on CPU; dead-end on GPU for FD
  specifically; sim-side GPU exists but means a different simulator."

  ON THE FASTDOWNWARD SIDE — substantial prior work exists:
    - HDA* / Distributed Fast Downward (Kishimoto, Fukunaga, Botea —
      ICAPS 2009; arXiv 1201.3204) — Hash-Distributed A* as a drop-in
      replacement for FD's sequential A*.  MPI-based; works on shared
      memory, distributed, or mixed.  30-60x speedup on 128 cores.
      Uses Abstract Zobrist Hashing to assign each state to a unique
      process for local duplicate detection.  Live code:
      github.com/jinnaiyuu/distributed-fast-downward (research code,
      not tracking modern FD; targets OPTIMAL A*, not the satisficing
      greedy best-first that PDDLStream's adaptive algorithm uses —
      so a drop-in port to our pipeline is non-trivial).
    - ArvandHerd (IPC 2011 winner, parallel track) — 4 Arvand configs
      + 1 LAMA config running on separate processors with minimal
      cross-process comms.  This is exactly the "FD portfolio mode"
      angle; it works because the "find a meaningful split" problem
      is hard and config-portfolio is the practical workaround.
    - Fast Downward Stone Soup (Helmert et al., ICAPS 2011) and
      Fast Downward Remix — sequential portfolio configurators
      (15 of 38 FD configs, varying time slices).
    - Springer 2021 "Parallel best-first search algorithms for
      planning problems on multi-core processors" — async work-pool
      design with abstraction-based duplicate-detection partitioning;
      maps directly to a "split by CPU thread count" framing.
    - arXiv 2407.21485 (2024) — Parallel Strategies for Best-First
      Generalized Planning; recent active research.

  ON THE GPU SIDE — essentially dead-end for classical PDDL planning:
    - FD's data structures are pointer-heavy and not trivially
      vectorisable; the literature has very few wins.  The most
      likely win is heuristic batching (e.g. pattern-database
      lookups) — narrow, not a general parallelisation.
    - LLM-generated heuristics on GPU (arXiv 2501.18784, 2025) is
      the more promising angle: don't parallelise the search,
      parallelise the heuristic computation.

  ON THE SIMULATOR SIDE — PyBullet GPU is dormant; alternatives
  exist but mean swapping simulator entirely:
    - Bullet 3.x has an OpenCL GPU rigid-body pipeline (Erwin
      Coumans, 2012) — 100k bodies real-time on a Radeon 7970.
      Never wired into PyBullet; maintainer position on GitHub
      issue #3105 is "may be enabled in the future."  Code dormant
      since AMD stopped funding the work in ~2013.
    - For GPU-accelerated robotics sim today: NVIDIA Isaac Sim /
      Isaac Lab (thousands of envs in parallel, Omniverse-based);
      MuJoCo MJX (JAX); Genesis.  All require a full simulator
      swap — out of thesis scope.
    - Within PyBullet itself, the realistic CPU-parallelism win
      is rayTestBatch numThreads — see #69 for the 6-call-site
      one-line patch.

  WHY THIS STAYS FOR LATER (not promoted to a numbered issue):
    - Even if HDA*/MPI was wired into the planner stack, our
      wall-clock perf complaint is upstream of FD search itself
      (see #50/#62 — Python-side stream/init redundancy).  An
      infinitely-fast FD would shrink an already-small slice.
    - The realistic CPU-side win that doesn't require any
      simulator/planner swap is captured concretely as #69
      (rayTestBatch numThreads).
    - GPU is research-grade work measured in months, not days,
      and the literature is thin enough that it's a dissertation
      topic in its own right — not a pre-submission tweak.

  Cross-references: #50/#62 (Python-side perf hunt — same goal,
  different layer); #69 (rayTestBatch numThreads — the realistic
  spinoff).

What:  Three coupled commits today (7116e73 / 65146ad / 8774e6f /

       (3-11 deg), JOINT_FIXED preserves tilt through move, place

           cube moves with the EE — the cube comes UP 15 mm with
           the gripper.  At release the cube falls 15 mm onto
           the table.  bottom_z at verify will be at table_z

           expected_support_z=table_z still applies.
