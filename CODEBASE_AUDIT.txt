CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

META-ISSUE — OLD #95 SYSTEMIC PDDL STATE DRIFT (dissolved)
       The 4 drift channels between PDDL symbolic state and PyBullet

#36. HARDCODE SMALL POST-RELEASE / POST-GRASP LIFT (PLACE & STACK PRIMARILY, PICK COSMETIC)
================================================================================
Priority: TIER 3 — Robustness workaround for motion-planning fragility
Where:    execution.py execute_place (~line 605, just before reading
          actual_joints — primary target),
          execution.py execute_stack (~line 710, mirror of execute_place
          — primary target, even more important because the freshly
          stacked cube sits on a tall support),
          execution.py execute_pick (~line 499, optional cosmetic
          add-on so the holding-goal terminate-at-contact does not look
          weird).
Depends:  None
Effort:   <0.5 day

What:  After the 2026-04-22 execution refactor, none of execute_pick /
       execute_place / execute_stack lift the arm before returning.
       Every action ends at its contact pose:
         - pick:  arm at object centre, fingers wrapped, cube hovering
                  near the table.
         - place: arm at release pose, gripper open, EE sitting on top
                  of the just-placed cube.
         - stack: arm at release pose, gripper open, EE sitting on top
                  of the freshly stacked cube on top of the support.

       Why the lift matters MOSTLY for place / stack:
         The next planned `move` action's plan_motion runs from
         wherever the arm currently sits.  After place/stack that
         start config is "EE in contact with a cube the planner
         hasn't accounted for as an obstacle yet" — the motion
         planner is fragile here ("shotty motion planning" per
         supervisor).  Stack is the worst case: the EE starts on top
         of a tall column and any motion has to clear the column.
         A small Z-lift before returning gives plan_motion safe
         headroom and is the pragmatic workaround.

       Why the lift is COSMETIC for pick:
         After pick the next `move` is in free space (the cube has
         become attached to the EE; everything beneath is what the
         planner already saw).  The terminate-at-contact for the
         holding goal is visually weird (cube near table) but does
         not create a follow-on motion-planning hazard.  Add the
         lift here only if visual polish matters.

       Earlier proposal (DROPPED 2026-05-06):
         Add (at_config q_home) to the holding goal so the planner
         chains pick → move(home), and the second move's plan_motion
         does the lift "for free" with no execution-side hidden
         motion.  Principled / one-to-one-with-PDDL.  Skipped per
         supervisor sync 2026-05-06 because the existing motion
         planner is too flaky to rely on for the second move;
         a hardcoded execution-side lift is the pragmatic answer.
         Supervisor's framing: "this is not cheating, it's a
         workaround for the shotty motion planning."

Fix:   ONE COMMIT — common helper + three call sites:
       (a) Add a private helper in execution.py:
             def _apply_post_action_lift(robot_id, contact_ee,
                                         orientation, contact_joints,
                                         pc, gui,
                                         lift_height=0.10) -> joints:
               lift_ee = contact_ee + [0, 0, lift_height]
               lift_joints = solve_ik(robot_id, lift_ee, orientation,
                                       pc, seed=contact_joints)
               if lift_joints is None: return contact_joints  # fail-safe
               move_robot_smooth(robot_id, lift_joints, gui)
               return lift_joints
           If lift IK fails, fall through silently to the contact
           config — never abort the surrounding pick/place/stack just
           because the cosmetic lift can't reach.
       (b) Call it in execute_place AFTER the 30-step settle and
           BEFORE reading actual_joints — so final_config reflects
           the lifted pose for the next move's plan_motion seed.
       (c) Call it in execute_stack AFTER the 60-step settle and
           BEFORE reading actual_joints — same reasoning, taller
           support means the lift matters more.
       (d) Optional: call it in execute_pick AFTER createConstraint
           and BEFORE reading actual_joints, with a smaller lift
           (~5 cm).  Cosmetic only; skip if it complicates the
           refactor invariant.
       (e) Read actual_joints AFTER the lift in every call site
           (audit #86, current_config tracking).
       (f) Update each docstring to note the hardcoded execution-side
           lift as an accepted thesis simplification (see
           THESIS_NOTES.md §19).

       NB: this is a deliberate departure from the refactor's
       "execution one-to-one with PDDL actions" invariant.  The lift
       is invisible to the planner — that is the whole point — so
       the next move's plan_motion seed (final_config) carries the
       lifted pose forward.

Related: execution.py refactor 2026-04-22 (originally REMOVED the lift —
         this restores it with a different rationale and at the
         place/stack sites that actually need it),
         THESIS_NOTES.md §19 (entry for the workaround),
         #49 (tray goal: every successful run ends with place or
              stack, so this lift directly improves the tray-goal
              evaluation experience).

#41. STACK GOAL DOES NOT CONSTRAIN BASE TO BE ON THE TABLE  [DONE 2026-05-07]
================================================================================
Priority: TIER 1 — Semantic completeness of the goal language;
          user-prioritised 2026-05-06 to top of the pre-evaluation
          roadmap.  Reasoning: the (on a b) chain alone admits
          floating-tower interpretations that the verifier (#40)
          cannot rule out without an on_table clause; making the
          goal explicit is the single cheapest cleanup of the goal
          language before #49 (tray goal) builds on the same
          predicates.
Where:    test_full_pipeline.py (build_stack_goal ~line 111),
          pddl/domain_pddlstream.pddl (new predicate or convention)
Depends:  None.  Could ride with #40's verifier work.
Effort:   <1 day

What:  build_stack_goal produces goals of the form
       (and (on t_1 t_2) (on t_2 t_3) ... (on t_{H-1} t_H))
       with no clause asserting that t_H (the base) is on the
       table.  Currently this is moot: all cubes start on the
       table; no action lifts a cube off the table without
       replacing it somewhere; so the base is always table-resting
       by induction.

       However it is still a correctness gap:
       (a) If #39 is ever partially fixed but not consistently,
           pick/place sequences could leave the base floating
           (ungrounded stack).
       (b) A future plan shape (e.g. pick the base cube, sense
           under it, then re-stack) would let the base end up
           anywhere.
       (c) The goal is semantically incomplete: a reader cannot
           tell from the goal whether "a tower" is intended or
           just a chain of (on) relations in mid-air.

Fix:   ONE COMMIT — options:
       (a) Add an (on_table ?o) predicate to the domain, emit
           (on_table X) in init for every cube, and have
           pick/place/stack effects maintain it (pick removes,
           stack removes, place emits).  build_stack_goal adds
           (on_table t_H) to the conjunction.  Verifier (#40)
           extends to this clause.
       (b) Represent the table as an Obj (with is_table flag)
           and use (on t_H table) with a is_table constant.  More
           symmetric; bigger domain change.
       Option (a) is lighter-touch and compatible with the
       existing domain.

Related: #40 (physics verifier should check the on_table clause),
         #30 (stack feature that needs this completion),
         pddl/domain_pddlstream.pddl comment at ~line 64:
           "Objects not mentioned in any (on ?o ?x) fact is treated
            as table-resting" — this convention becomes explicit.

#48. PICK DOES NOT CLEAR on_relations[X] AFTER REMOVING X FROM A STACK  [USER-PRIORITISED 2026-05-06; BUNDLE WITH #41]

          goal) makes pick→place(tray) plan shapes normal, and any
          target that starts on top of another cube would trigger the

Where:    test_full_pipeline.py pick handler (~line 891-946),
          on_relations dict (~line 299)
Depends:  None.  Independent of #40 (which gates the WRITE side; this is

Effort:   <0.5 day

What:  The PDDL pick action's conditional effect

         (forall (?x)
           (when (on ?o ?x)
             (and (not (on ?o ?x)) (clear ?x))))

       symbolically clears (on ?o ?x) for any support ?x.  The Python
       on_relations dict — passed to planner.plan() and read by
       goal_satisfied() — is NOT cleared.  After picking X off Y,

           future plan shape leaves a held cube but the goal contains
           a stale (on X Y) clause that hasn't been refuted.
       (c) Symmetric to #40 — same "executor-tracked symbolic state
           goes stale vs. physics" theme.

Fix:   ONE COMMIT — in the pick handler, after the *** PICKED UP ***
       log line, clear obj_str's entry:

         on_relations.pop(obj_str, None)

       No domain change.  Place handler unchanged (place doesn't write
       on_relations).  Stack handler unchanged (#40's gate already
       handles the write side).

Related: #40 (write-side gate; this is the clear-side counterpart),
         #47 (atoms-vs-physics theme),
         pddl/domain_pddlstream.pddl pick conditional effect.

#50. PLANNER PERFORMANCE INVESTIGATION — REDUNDANCY HUNT
================================================================================
Priority: TIER 2 — Affects every eval-run wall-clock time;
          user-flagged 2026-05-06 ("planner is too slow").
Where:    pddlstream_planner.py (_build_init, planner.plan, stream
          stitching), streams.py (test/gen streams; per-call cost),
          execution.py (between-replan re-grounding paths), and the
          adaptive search inner loop (FastDownward grounder).
Depends:  None (independent investigation).  Pairs naturally with
          #9 / #12 — any wins should show up in the scalability plots.
Effort:   1 week of focused profiling, then variable per-redundancy fix.

What:  User raised at 2026-05-06 sync:
         "Personally I think the planner is too slow and want to find
          out ways to make it faster or at least discover and remove
          some big redundancies."

       This is broader than #30 (which targeted one specific
       conditional-effect cost in the pick action and is now
       [WONTFIX]).  #50 is a profiling-driven sweep — find ALL the
       big redundancies, regardless of whether they live in the
       grounder, the streams, or the Python init-fact construction.

       Known suspects (no measurement yet — confirm with profiler
       before fixing):
       (a) _build_init re-emits ALL static facts every replan.  Many
           of them (boxel_fits, semantics, geometry-derived
           predicates) are invariant under sense / pick / place and
           could be cached across replans of the same run.  Cf.
           archive note "BOXEL_FITS / STATIC INIT FACTS — REBUILT
           EACH REPLAN" — that note explained WHY the test-stream
           variant was abandoned, but did not rule out caching the
           Python-side computation.
       (b) _compute_placement_view_blocks runs every _build_init at
           O(S × 25 × F) ray-AABB tests (#34 audit).  Cache by
           (shadow geometry, free-boxel layout) hash; invalidate only
           when reboxelize_free_space mutates the registry.
       (c) The placement sampler in streams.py may be re-sampling
           geometrically equivalent poses; check if pose-level
           dedup would help.
       (d) `test_*` streams that fire many times per skeleton against
           Python-side data that doesn't change between calls — fold
           the test result into init facts and drop the stream.
       (e) Any per-action Python loop in execute_* that scales with
           registry size (look for O(N^2) over object/shadow lists).
       (f) FastDownward grounder cost: how much grounded-state size
           is inflated by predicates that have only a handful of
           true ground atoms at runtime?  candidate for
           negative-precondition / typing tightening.

Fix:   APPROACH (no single commit — investigation first):
       (a) Land a profiling pass: cProfile around planner.plan() and
           around _build_init separately; aggregate over a fixed seed
           sweep on the default scene and on scalability_scene (small
           and large).  Output a Python-side flame graph + a
           FastDownward translator-time / search-time / total-time
           breakdown (the translator already prints these; capture
           them into timing_summary.json).
       (b) For each hotspot ≥ 10% of total time, decide:
             - Cache (input is deterministic and stable across replans)
             - Vectorise / numpy-ify (pure-Python ray loops, etc.)
             - Drop / replace (dead code path or redundant fact)
       (c) Land each redundancy fix as its own commit with a before-
           after timing entry in the commit message + a one-line
           note in THESIS_NOTES.md "evaluation methodology" if it
           changes the per-call time used in the Results chapter.
       (d) Re-run the seed sweep after each fix to make sure the
           change does not regress success rate.

Care: Do not start optimising before profiling.  The planner has many
       plausible-looking suspects (init build, streams, grounder) and
       only one of them is likely to dominate at any given scene size.
       Premature optimisation will make the code harder to read for
       little wall-clock gain.

Related: #9 (experiment runner — needed to gather timing samples),
         #12 (scalability eval — wins land here visibly),
         #30 [WONTFIX] (the per-action conditional-effect cost is one
              specific instance of the broader question — out of
              scope per supervisor; revisit only if profiling shows
              it dominates after other wins),
         #34 (placement-view-blocks ray-AABB cost — concrete suspect),
         FOR LATER bullets (incremental boxel recomputation, dense
              lattice — both touch planner cost).
