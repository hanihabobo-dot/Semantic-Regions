CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

RUNNING THE PIPELINE — POWERSHELL / WSL WRAPPERS
       Machine-specific run instructions, moved out of README.md on
       2026-05-18 per user direction (README and wiki must stay
       machine-agnostic).  Kept here because this is the only record
       of the launch wrappers — they live in the user's PowerShell
       profile, not in the repo.

       The repo runs in the wsl_env/ WSL venv (see entry above) with
       PDDLStream vendored as a sibling checkout, ../pddlstream_lib.
       Three wrapper functions in $PROFILE
       (Microsoft.PowerShell_profile.ps1) launch it from PowerShell.
       <repo> below = /mnt/c/Users/HaniAlassiriAlhabbou/git/Semantic_Boxels
         run_boxels  ->  cd <repo> && source wsl_env/bin/activate &&
                         DISPLAY=:0 LIBGL_ALWAYS_SOFTWARE=1
                         PYTHONPATH=<repo>/../pddlstream_lib
                         python3 test_full_pipeline.py $args
         run_eval    ->  cd <repo> && source wsl_env/bin/activate &&
                         PYTHONPATH=<repo>/../pddlstream_lib
                         python3 eval_runner.py $args
         plot_eval   ->  cd <repo> && source wsl_env/bin/activate &&
                         python3 eval_plotter.py $args

       One-time wsl_env bootstrap (wsl_env/ is gitignored):
         wsl bash -lc "cd <repo> && python3 -m venv wsl_env &&
           source wsl_env/bin/activate &&
           pip install pybullet numpy matplotlib"
       matplotlib is required by eval_plotter.py — without it the
       plotter prints text tables instead of PNGs (look for the
       "(matplotlib not available)" marker in stdout).

       plot_eval: pass the /mnt/c/... WSL form of the aggregated.csv
       path.  PowerShell mangles backslashes when a Windows path has
       escape-like substrings (\U, \H, ...).  Three PNGs land next
       to the CSV.

       PowerShell pitfalls:
       - "[plotter] not found: C:Users..." (no backslashes) — a
         Windows path lost its backslashes inside a double-quoted
         "$(...)"; use the /mnt/c/... path or Join-Path as one arg.
       - "(matplotlib not available)" — wsl_env/ lacks matplotlib;
         re-run the bootstrap pip install.
       - "bash: ...conda.sh: No such file or directory" — the shims
         activate the wsl_env venv, not anaconda; install into the
         venv, not base.

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

#95. STACK-SCENE CELL-TAG MISNAMING — occN/tgtN TOKENS DON'T MATCH run_config
================================================================================
Status:   OPEN.  Filed 2026-05-20 from the #93 sweep review.
Priority: TIER 3 — data-hygiene / interpretability.  Doesn't
          affect correctness; affects the eval chapter's
          ability to plot stack scalability honestly and makes
          per-cell debug confusing.
Where:    eval_runner.py::cell_tag (the stack branch of the
          tag formatter) and the MATRIX_PRESETS reuse of
          n_occluders as the stack-height axis (around the
          SCALABILITY_VS_TIME definition).
Depends:  none.
Effort:   0.5-1 day.

What:  SCALABILITY_VS_TIME matrix sweeps n_occluders ∈ {2,3,4}.
       For the random-pairs goals (holding, find-and-tray-
       stack) this is honest — run_config.n_occluders matches
       the cell-name occ token.  For the stack goal it is not:
       verified, the occ2 stack cell -> run_config
       n_occluders=0, n_targets=3, stack_height=2; the occ4
       stack cell -> n_occluders=0, n_targets=4, stack_height=4.
       The stack scene ignores the matrix occluder value and
       re-interprets the swept value as stack height.  The
       cell-name occ{N}/tgt{N} tokens then no longer correspond
       to anything in the run.

       Consequence:
       - Per-cell inspection: opening
         occ4_tgt3_hid0_..._stack_..._semantic and finding
         n_occluders=0 / n_targets=4 in run_config is
         confusing.
       - Aggregation: a naive group-by occ-from-cell-name
         aggregates DIFFERENT stack difficulties together
         (stack_height 2 vs 4) — gives wrong scalability
         conclusions.
       - The thesis chapter's "success vs n_occluders" curve
         cannot include the stack goal; it varies stack height
         instead.

Fix:   Option A (minimal, recommended first): in eval_plotter
       (and any downstream parser) read difficulty from
       run_config — stack_height for stack cells, n_occluders
       for the random-pairs goals.  Update plot captions to say
       which axis is shown.  Document the difficulty axis per
       scene in the eval-chapter writeup.
       Option B (renaming, more invasive): extend cell_tag to
       use scene-relevant tokens (stack_h{N}_ for stack cells;
       random-pairs already uses randpairs_occ{N}_).  Avoids
       legacy-name confusion but breaks any existing tooling
       that parses cell names.
       Option C: A in the short term, B in the next sweep
       generation.

Care:
       (1) preview_plots.py (#93) currently buckets stack as
           one group without slicing by stack_height.  A
           proper §5 plot needs the per-stack-height
           breakdown — gate on Option A.
       (2) MATRIX_PRESETS for SCALABILITY_VS_TIME keeps
           n_occluders as the axis NAME even when the stack
           scene re-interprets it.  Add a comment at the
           preset definition so the next reader does not
           conclude it is an actual occluder sweep on stack.
       (3) Already-aggregated.csv from prior sweeps has the
           same naming; aggregations re-run after Option A
           will reclassify stack rows by stack_height.

#97. DESIGN A FINE-RESOLUTION DISCRIMINATING SCENE — DEMONSTRATE WHERE mbs0.05 / FINER PARTITION ACTUALLY HELPS
================================================================================
Status:   OPEN.  Filed 2026-05-20 in response to #93's finding
          that semantic and semantic+mbs0.05 are statistically
          indistinguishable on the current sweep — the
          headline #77 "finer resolution wins under a fixed
          time budget" hypothesis is unsupported on the
          scenes the eval actually exercises.
Priority: TIER 1 — gates #77's headline claim and feeds the
          thesis evaluation-design discussion
          (thesis_audit.md #140).
Where:    test_full_pipeline.py scene builders +
          boxel_env.py; collect_eval_corpus.py for the new
          scene's corpus; MATRIX_PRESETS in eval_runner.py
          for the sub-matrix that exercises it.
Depends:  #93 (null-result finding); #77 (parent anytime
          sweep).
Effort:   3-5 days for design + scene implementation +
          corpus probe + a smoke sub-sweep, before
          re-running at corpus scale.

What:  #93 shows semantic and semantic+mbs0.05 are
       indistinguishable on every goal: stack identical
       seed-for-seed at 184/300 each; holding 42.3% vs
       46.3% (within seed noise); find-and-tray-stack
       39.7% vs 32.3% (mbs0.05 actually WORSE).  The
       0.05 m leaf floor does not change outcomes because
       - objects are ~4 cm cubes;
       - semantic's auto_cell floor (audit #67) is
         ~6-9 cm (largest object's footprint + 1 cm) and
         dominates the merged partition;
       - CellMerger collapses same-label free-space into
         large boxels regardless of leaf size;
       - OBJECT and SHADOW boxels are per-object and
         never touched by --min-boxel-size.
       The current scenes therefore do NOT exercise the
       regime in which a sub-auto_cell leaf could matter.
       #77's hypothesis is not necessarily wrong — the
       eval does not test it.

Fix:   Design and implement at least one task / scene
       where ONLY a finer-than-auto_cell free-space
       partition can solve the problem.  Candidate
       patterns:

       (a) Narrow-corridor reach — two large occluders
           forming a sub-auto_cell-width gap with a target
           reachable only by routing the arm through the
           gap.  Coarse partition cannot represent the
           corridor as free; finer partition can.
       (b) Sub-cell pocket placement — place an object
           into a slot whose entry is smaller than the
           largest object's footprint (a notched tray).
           Coarse cells reject the placement; fine cells
           admit it.
       (c) Sub-cell hidden target — a hidden target whose
           shadow is smaller than auto_cell, lodged
           between two occluders.  Coarse partition cannot
           isolate the shadow from the occluders for
           sensing.
       (d) Dense small-occluder shelf — many sub-auto_cell
           occluders; reasoning about reach and visibility
           requires fine free-space partition.

       Steps (recommend (a) for the first pass — it has
       the cleanest reach vs visibility trade-off):
       Step 1 — pick one pattern.
       Step 2 — implement the scene builder + register it
                in test_full_pipeline.py's scene dispatch.
       Step 3 — extend collect_eval_corpus.py with a
                probe for the new scene; produce a small
                corpus (e.g. 50 seeds).
       Step 4 — add a SCALABILITY_VS_TIME sub-matrix using
                the new scene at the {auto, 5 cm}
                resolution axis already swept.
       Step 5 — smoke sub-sweep (~50 seeds).  Verify
                discrimination: semantic-auto should fail
                or be slow, semantic+mbs0.05 should
                succeed, uniform should fail (auto-clamped
                per #66).
       Step 6 — once discrimination is shown, expand to
                full corpus and re-run.

Care:
       (1) The discriminating scene must be HONEST: a
           realistic manipulation task, not a contrived
           needle-threading toy.  Reviewers reject
           artificial geometry built only to embarrass
           the baseline.
       (2) For the comparison to mean anything, uniform
           must genuinely fail (or be very slow) on the
           new scene.  If uniform can solve it at its
           auto_cell, the test is not isolating
           resolution.  Likewise semantic-auto must fail —
           if both auto-config planners solve it, mbs0.05
           still adds nothing.
       (3) #93's null result on the current scenes must
           remain in the chapter even if Step 5 succeeds.
           The story is: "in the regime our scenes
           exercise, leaf floor is irrelevant; in this
           new scene it is decisive."  Both findings
           stand.
       (4) Match the new scene's max-plan-time to the
           rest of the sweep (1800 s) for fair anytime
           curve comparison.

Related:
       #77 [OPEN, TIER 1] — the parent anytime sweep;
            this scene is what its hypothesis actually
            requires.
       #93 — null result that prompted this issue.
       #95 — stack-scene cell-tag misnaming (the new
            scene must not repeat it).
       thesis_audit.md #140 — the mirror evaluation-
            design issue on the thesis side.

#98. COARSER-THAN-AUTO_CELL FOLLOW-UP SWEEP — CHARACTERISE THE RESOLUTION AXIS ABOVE auto_cell
================================================================================
Status:   OPEN.  Filed 2026-05-20 as the symmetric counterpart of
          #97.  #93's null finding (semantic == semantic+mbs0.05
          at this scene scale) established that auto_cell is the
          binding constraint on the FINE end.  This issue probes
          the opposite direction — what happens when
          min_boxel_size is held strictly ABOVE auto_cell (1.5x,
          2x).
Priority: TIER 2 — informational; characterises the resolution-
          vs-success curve and feeds the Discussion (thesis #125
          / #140).  Not blocking the main #77 anytime claim, but
          gives the chapter a defensible "we swept resolution in
          both directions" story.
Where:    eval_runner.py MATRIX_PRESETS — extend
          SCALABILITY_VS_TIME's min_boxel_size axis (or sibling
          preset SCALABILITY_VS_RESOLUTION) with two coarse
          points.  test_full_pipeline.py effective_cell logic
          already supports mbs > auto_cell verbatim post-#77
          step 3a (the clamp was lifted in both directions).
Depends:  #77 step 3a [DONE]; #93 (the null finding this is the
          dual of); #95 (stack cell-tag misnaming, so stack rows
          stay interpretable).
Effort:   1-2 days for matrix extension + smoke sub-sweep +
          full re-run + plotter check.  Corpus is reusable
          (still random-pairs + stack scenes, same seeds).

What:  #97 asks "engineer a scene where FINER than auto_cell
       wins."  This is the dual: as min_boxel_size grows ABOVE
       auto_cell, at what point does coarser representation
       start to hurt?  Today the sweep covers two points (auto,
       and mbs=0.05 which falls AT or below auto_cell for these
       scenes); neither exercises the coarse end of the curve.

       Hypothesis: success rate falls and per-call planning
       time falls as min_boxel_size grows past auto_cell ---
       fewer free-space boxels means the planner has fewer
       atoms to reason over (cheap planning) but loses the
       representational fidelity needed for reach corridors,
       placement targets, and shadow disambiguation (harder
       to find a plan, more plans may not exist).  The cross-
       over is what the sweep reveals.

       Expected reading (random-pairs auto_cell ~ 9 cm; stack
       auto_cell ~ 6 cm):
       - auto_cell                — current baseline (#93:
                                    semantic 61% stack /
                                    42% holding / 40% f-a-t-s).
       - 1.5x auto_cell           — coarse but still finer than
                                    the largest object.
                                    Plausible: modest success
                                    drop, modest speed-up.
       - 2.0x auto_cell           — approaches the uniform-cell
                                    scale.  Might match
                                    uniform's success on stack
                                    (~65%) but with semantic's
                                    object/shadow boxels intact,
                                    so init-facts stay an order
                                    of magnitude below uniform.

Fix:   Steps:
       Step 1 — add the coarse-end axis to SCALABILITY_VS_TIME.
                Two new min_boxel_size points (1.5x and 2.0x
                auto_cell) per scene.  Because auto_cell varies
                per scene, either:
                  (a) compute the multiplier inside test_full_-
                      pipeline.py from a CLI flag like --mbs-
                      ratio 1.5; or
                  (b) hard-code per-scene numerics in the
                      matrix sub-matrices: random-pairs
                      {0.135, 0.18} m, stack {0.09, 0.12} m
                      (or whatever the measured auto_cell is at
                      the time of the sweep).
                (b) is simpler in code and easier to audit; (a)
                is cleaner if the scene set grows.
       Step 2 — smoke sub-sweep (5 seeds per cell, ~120 cells)
                to verify the mbs > auto_cell path runs.
                Confirm run_config.effective_min_boxel_size
                shows the requested coarse value verbatim and
                CellMerger doesn't accidentally fold every
                free-space cell into one giant boxel.
       Step 3 — full corpus sweep (100 seeds) once smoke is
                clean.
       Step 4 — eval_plotter: solved_vs_time.png already groups
                by (baseline, min_boxel_size), so new lines
                appear automatically.  Confirm the per-goal
                legend stays readable with 4-5 series; if not,
                consider a colour gradient over the semantic mbs
                values (auto = blue, 1.5x = light orange, 2x =
                dark orange) keeping uniform gray.
       Step 5 — chapter writeup: the resolution-vs-success
                curve at FOUR points (mbs<<auto via #97,
                mbs==auto, 1.5x, 2x) characterises where
                semantic's adaptive partition gains and loses.
                Even if #97 doesn't land, this issue alone gives
                the chapter a resolution-axis sweep.

Care:
       (1) auto_cell varies per scene (~6 cm stack, ~9 cm
           random-pairs).  A single matrix-level mbs constant
           means the effective multiplier differs across
           scenes.  Either accept that and document, or use
           per-scene presets / a --mbs-ratio flag.
       (2) The audit-#77 step 3a clamp lift was tested
           primarily at SMALLER-than-auto values.  Smoke must
           also verify the larger end: that the free-space
           octree does subdivide above auto_cell, that
           CellMerger respects label boundaries, and that
           planner streams can still ground reach / place /
           sense at coarser cells.  No code change anticipated,
           but the path is less exercised.
       (3) If 1.5x and 2x produce the same success / planning
           time as auto, the resolution axis is flat above
           auto_cell at this scene scale — itself a finding.
           Report it and stop the sweep there rather than
           expanding the axis further.
       (4) The coarse-end variant cells inherit #95's stack-
           cell-tag misnaming.  Aggregate stack rows by
           stack_height (run_config), not by the occ token.

Related:
       #77 [OPEN, TIER 1] — parent anytime sweep; this is the

       #97 — fine-end discriminating scene; this issue is the
            coarse-end probe.
