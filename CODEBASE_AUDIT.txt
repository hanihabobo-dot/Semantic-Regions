ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET GUI FREEZES DURING PLANNING — THIS IS NORMAL
       The RenderingLock (robot_utils.py) disables COV_ENABLE_RENDERING
       for the entire planning phase (test_full_pipeline.py ~line 346,
       audit #60).  While planning runs (often 30-130 s), the PyBullet
       window appears frozen — no zoom, pan, or rotation works.
       The GUI unfreezes when planning finishes.
       If planning takes very long, WSLg's Weston compositor may drop
       the X11 connection, crashing the process with:
         XIO: fatal IO error 11 (Resource temporarily unavailable) on X server ":0"
       Fix: restart WSL (wsl --shutdown from PowerShell, then reopen
       terminal).  Keyboard camera controls (R/F for zoom, WASD for
       pan) also do not work while rendering is locked.

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

STATIC GEOMETRY IN INIT (e.g. boxel_fits) — FUTURE MERGE/SPLIT
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot (old #102).  They are valid only for that partition and
       geometry for one planning problem.

       If we ever **re-split or re-merge** free-space (or other) boxels
       **after** an action — e.g. online refinement between replans — the
       registry and AABBs can change.  Then every init fact that depends on
       boxel identity or extents (not only boxel_fits, but also is_free_space,
       Boxel objects, etc.) must be **rebuilt from an updated registry** before
       the next solve: effectively **re-initialise** the PDDLStream problem
       (or an equivalent full refresh).  Do not assume a prior init stays
       valid across merge/split.  Overlaps the re-boxelization requirement
       in #4; call this out again when implementing dynamic partition updates.

BOXEL_FITS (old #102) — STATIC INIT, NOT A TEST STREAM
       The place precondition (boxel_fits ?o ?b) is satisfied by init facts
       only.  BoxelStreams.test_boxel_fits() runs once per problem in
       _build_init() for each (object, free_space) pair that geometrically
       fits.  An earlier implementation used a PDDLStream test stream
       (test-boxel-fits); adaptive search re-evaluated it heavily across
       skeletons and dominated runtime despite a cheap per-call check.
       stream.pddl no longer declares that stream; domain and predicate are
       unchanged.

#1. EXECUTION SUB-MOVES ARE COLLISION-BLIND
================================================================================
Priority: TIER 0 — Start here (most self-contained)
Where:    test_full_pipeline.py (execute_pick ~line 782, execute_place ~line 894)
          robot_utils.py (move_robot_smooth)
Depends:  None
Effort:   1-2 days

What:  The sub-moves inside execute_pick (approach, descend-to-contact,
       lift) and execute_place (approach, descend, retreat) all use
       move_robot_smooth(), which is purely linear joint-space
       interpolation with NO collision checking.

       The PDDL planner calls plan_motion() (full RRT-Connect collision
       checking) only for the symbolic move action.  Everything inside
       execute_pick/execute_place runs blind at execution time.

       Consequence: the robot collides with other objects during
       approach/retreat.  The hardcoded heights (approach 0.10 m,
       lift/retreat 0.25 m) are never validated against the scene.
       This is the observed failure: "robot struggles with move to
       place path."

Fix:   ONE COMMIT — replace move_robot_smooth() calls with
       collision-aware motion:
       (a) In execute_pick: for the approach and lift segments, call
           BoxelStreams.plan_motion() or is_path_collision_free() to
           get a collision-free trajectory, then follow its waypoints.
       (b) In execute_place: same for approach and retreat segments.
       (c) Pass ignored_body_ids (grasped object during pick approach;
           none during retreat) matching what plan_motion already does
           for the PDDL move action.
       The descend-to-contact segment can remain linear (short vertical
       move in collision with the target object by design).

       Reuses existing RRT/collision infrastructure with minimal new
       code and NO PDDL domain changes.

Related: #7/#8 (full PDDL decomposition is the long-term fix)

#3. SENSE ACTION HIDDEN MOVE — NO at_config PRECONDITION
================================================================================
Priority: TIER 0 — Quick PDDL consistency fix
Where:    domain_pddlstream.pddl (sense action, ~line 128),
          test_full_pipeline.py (sense handler, ~line 425)
Depends:  None
Effort:   1-2 days

What:  The sense handler secretly moves the robot to home
       configuration before raycasting (workaround for old #79: arm
       blocking camera).  The PDDL sense action has NO at_config
       precondition — the planner does not know sensing requires the
       arm to be out of the way.

       This causes PDDL state drift: after sense, the robot is
       physically at home but PDDL still records (at_config ?q) from
       the last move action.  If the planner chains sense -> pick,
       the pick starts from the wrong config in the symbolic model.

Fix:   ONE COMMIT — choose one approach:

       OPTION A (minimal): Add (at_config ?q_home) as a precondition
       to the sense action in domain_pddlstream.pddl.  Emit
       (at_config q_home) in _build_init() when the robot starts at
       home.  The planner will generate move -> sense sequences when
       the robot is not at home.  Remove the hidden move from the
       sense handler (the planner now handles it explicitly).

       OPTION B (pragmatic): Add a prepare_sense action that moves
       the robot to home.  The sense action requires (at_config
       ?q_home) as a precondition.  prepare_sense's effect sets
       at_config to q_home.  More explicit but adds a PDDL action.

       OPTION A is recommended — simpler, and move already exists.

Related: old #79 [DONE] (arm blocks sensing), old #86 [DONE] (config
         tracking on replan), #18 (observational symptom: arm in view path)

#4. POST-ACTION RE-BOXELIZATION — STALE PARTITION
================================================================================
Priority: TIER 1
Where:    test_full_pipeline.py (execution loop),
          boxel_data.py (BoxelRegistry),
          cell_merger.py
Depends:  None (re-merge uses existing convex merge; full merge #2 deferred)
Effort:   3-5 days

What:  After a pick-and-place action, the boxel partition is NOT
       recomputed.  Stale data includes:
       (a) OCCUPIED boxels that are now FREE (object picked up)
       (b) FREE boxels that are now OCCUPIED (object placed there)
       (c) Shadow regions that changed (occluder moved)
       (d) Merge structure is stale (needs re-run of convex merge pass)

       The planner operates on this stale partition, causing it to:
       - Try placing in no-longer-free boxels
       - Miss newly available free space
       - Plan sense actions for shadows that no longer exist

Fix:   ONE COMMIT — implement full re-boxelization (naive first):
       (a) After each scene-changing action (place), re-evaluate every
           boxel's occupancy status against current PyBullet state.
       (b) Recompute shadow regions for relocated occluders.
       (c) Re-run the merge pass (convex-only; stricter merge deferred
           — archive/CODEBASE_AUDIT_DEFERRED.txt #2).
       (d) Rebuild shadow_occluder_map.
       (e) Feed updated BoxelRegistry into _build_init() for next
           planning cycle.

       CRITICAL: observation history (which boxels have been
       sensed / known-empty) MUST survive re-boxelization.

       Incremental recomputation is in FOR LATER.

Related: old #44 [ACCEPTED] (stale shadow AABBs — subsumed by this),
         old #73 [DONE] (stale shadow_occluder_map — extended by this)

#7. PICK ACTION — PDDL/PYTHON SCOPE MISMATCH
================================================================================
Priority: TIER 2 — PDDL redesign, optional for thesis
Where:    domain_pddlstream.pddl (pick action, ~line 178),
          stream.pddl, streams.py, test_full_pipeline.py (execute_pick)
Depends:  None (but #1 provides a lighter-weight mitigation)
Effort:   ~1 week

What:  The PDDL pick action is modeled as an instantaneous single-
       config operation.  The Python execute_pick() is a 6-step
       choreographed sequence:
         1. open_gripper
         2. move to approach (IK solved at execution time)
         3. descend to contact (plan config)
         4. close_gripper
         5. createConstraint (weld object to gripper)
         6. lift (IK solved at execution time)

       Hidden sub-actions the planner cannot reason about:
         3 moves, 3 IK solves, 1 open_gripper, 1 close_gripper,
         1 createConstraint

       Consequences:
       - Planner cannot detect collisions on approach/retreat (#1
         provides a lighter-weight fix for this)
       - Execution-time IK solves can fail (handled by abort-and-
         replan, but planner has no awareness)
       - Post-pick config differs from PDDL's ?q (accepted
         simplification — old #62)

Fix:   ONE COMMIT — decompose pick into PDDL sub-actions:
       (a) approach_pick(?o ?b ?g ?q_approach)
       (b) grasp(?o ?g)
       (c) lift(?o ?q_lift)

       Each sub-action has its own preconditions and effects,
       including at_config updates.  Stream evaluations ~3x per pick.

       NOTE: #1 (collision-aware sub-moves) is the pragmatic fix.
       This full decomposition is architecturally correct but optional
       for thesis scope.

Related: #1 (lighter-weight collision fix), #8 (same pattern for place)

Effort:   ~1 week

       sequence:
         1. move to approach (IK solved at execution time)
         2. descend to contact (plan config)
         3. open_gripper
         4. removeConstraint
         5. 30 stepSimulation (settling)
         6. retreat (IK solved at execution time)

       Hidden sub-actions:
         3 moves, 3 IK solves, 1 open_gripper, 1 removeConstraint,
         30 stepSimulation

       (a) approach_place(?o ?b ?g ?q_approach)
       (b) release(?o ?b)
       (c) retreat(?q_retreat)

       Same reasoning as #7: #1 is the pragmatic fix; this is the
       architecturally correct long-term solution.

#14. SPARSE VISIBILITY CHECK — 8 AABB CORNERS MISS PARTIALLY VISIBLE OBJECTS
================================================================================
Priority: TIER 0 — Root cause of persistent sensing failures
Where:    boxel_env.py (oracle_detect_objects, ~line 719)
Depends:  None
Effort:   1-2 days

What:  oracle_detect_objects() determines whether an object is visible
       by casting exactly 8 rays from the camera to the 8 corners of
       the object's AABB.  If ANY ray reaches the object, it is
       visible; if ALL 8 are intercepted, the object is classified as
       invisible.

       8 corners is too sparse.  When an object is partially visible
       (e.g. a sliver sticks out between two occluders, or the top
       protrudes above a shorter occluder), none of the 8 corners may
       land in the visible gap.  The object is then classified as
       invisible despite being partially visible from the camera.

       Observed failure (run 2026-04-03): occluder_1 at (0.5, 0.2)
       was classified as invisible because all 8 AABB corner rays
       were intercepted by occluder_2 and occluder_3 in front.
       Consequences:
       (a) occluder_1 never entered the BoxelRegistry as an object
           boxel — the planner had no knowledge of it.
       (b) Shadow volumes from other occluders extended through
           occluder_1's position unsplit (#15).
       (c) The planner could never plan to relocate occluder_1,
           causing ALL subsequent sense actions to report 49-58%
           ray blockage.  10 plans executed, all failed — total
           mission failure.

Fix:   ONE COMMIT — increase ray density in oracle_detect_objects():
       (a) Replace 8-corner rays with a denser grid over the AABB
           surface (e.g. 4×4 grid per face = 96 rays, or N×N on the
           camera-facing face only).
       (b) Alternative: use the rendered depth buffer to check if
           any pixel corresponds to the object's body ID (segmentation
           mask approach — PyBullet's getCameraImage returns a
           segmentation buffer).  This is both cheaper and catches
           any visible pixel, no matter how small.
       (c) Either approach should use a visibility THRESHOLD: an
           object may be "visible" if ≥ K% of sample points reach
           it, to filter noise from sub-pixel slivers.

Related: #15 (shadow splitting depends on correct visibility),
         #16 (camera angle exacerbates this)

#18. PLAN SEQUENCE move→sense POSITIONS ARM IN CAMERA VIEW PATH
================================================================================
Priority: TIER 0 — Extends #3 (sense action design)
Where:    domain_pddlstream.pddl (sense / pick actions),
          test_full_pipeline.py (sense handler ~line 415)
Depends:  None (but #3 fix also addresses this)
Effort:   Included in #3

What:  The PDDL planner produces plans like:

         move(q_home, q_shadow, shadow_001) →
         sense(target, shadow_001) →
         pick(target, shadow_001, grasp, q_shadow)

       The move is a precondition of pick (needs at_config for the
       grasp configuration), and sense is also a precondition of pick
       (needs obj_at_boxel_KIF).  Since move and sense are
       independent, the planner linearizes them arbitrarily — and
       FastDownward consistently places move BEFORE sense.

       This means the robot arm physically travels TO the shadow
       region BEFORE the camera tries to see into it.  The sense
       handler compensates by secretly retracting the arm to home
       (line 428, audit #79), creating a wasteful round-trip:
         arm at home → move to shadow → retract to home → sense →
         (move to shadow again on next plan)

       Worse: if the home-retract is incomplete or the arm's home
       configuration still partially intersects the camera's FOV for
       certain shadow regions, sensing fails due to arm blockage.
       The sense_shadow_raycasting function correctly counts robot
       hits (line 701), but cannot distinguish this from occluder
       blockage in the plan-level feedback.

       The root cause is that sense has no at_config precondition
       (#3).  If it required (at_config q_home), the planner would
       be forced to order: sense → move → pick (or move_home →
       sense → move_to_shadow → pick), keeping the arm clear during
       sensing.

Fix:   Subsumed by #3.  Whichever option is chosen for #3, the
       resulting plan ordering will naturally avoid this problem.
       Documenting separately because the symptom (arm in view path)
       is observationally distinct from the PDDL state-drift symptom
       described in #3.

Related: #3 (same root cause and fix)

#21. BLOCKED SENSE CAUSES SHADOW SWITCHING AND FALSE "SEARCHED ALL" TERMINATION
================================================================================
Priority: TIER 0 — Control-loop correctness bug
Where:    test_full_pipeline.py (BeliefState, replanning loop, still_blocked handling)
Depends:  None
Effort:   1-2 days

What:  After a pick/place intended to clear a shadow, sensing can still return
       "still_blocked".  The execution loop then breaks to replan, but it does
       NOT preserve an obligation to keep clearing that SAME shadow.

       Current behavior:
       (a) belief.occluders_moved records the relocated occluder.
       (b) The blocked shadow remains unknown unless blocked_counts reaches 3.
       (c) The next plan is free to target a DIFFERENT unknown shadow instead
           of identifying and relocating the blocker that still obstructs the
           current one.
       (d) After 3 consecutive "still_blocked" outcomes on the same shadow,
           the loop marks that shadow as not_here even though the view was
           never clear and the shadow was never actually observed.
       (e) The run can then terminate with "Searched all shadows but target not
           found" even though some shadows were only blocked, not searched.

       This is different from #5.  #5 is about choosing a placement
       destination that re-blocks a corridor.  This issue is about the
       REACTIVE POLICY after sensing reports that a corridor is still blocked:
       the system switches search targets instead of continuing to clear the
       active shadow.

Fix:   ONE COMMIT — design decision needed:

       OPTION A — Introduce a belief/planning obligation to continue clearing
       the same shadow after still_blocked, until view_clear or clear_but_empty.
       OPTION B — Add a distinct shadow status such as blocked_unresolved;
       do NOT map it to not_here, and prioritize it before untouched shadows.
       OPTION C — On still_blocked, identify the current blocking object(s)
       for that same shadow and force replanning around one of those blockers.
