CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

RUNNING THE PIPELINE — POWERSHELL / WSL WRAPPERS
       Machine-specific run instructions, moved out of README.md on
       2026-05-18 per user direction (README and wiki must stay
       machine-agnostic).  Kept here because this is the only record
       of the launch wrappers — they live in the user's PowerShell
       profile, not in the repo.

       The repo runs in the wsl_env/ WSL venv (see entry above) with
       PDDLStream vendored as a sibling checkout, ../pddlstream_lib.
       Three wrapper functions in $PROFILE
       (Microsoft.PowerShell_profile.ps1) launch it from PowerShell.
       <repo> below = /mnt/c/Users/HaniAlassiriAlhabbou/git/Semantic_Boxels
         run_boxels  ->  cd <repo> && source wsl_env/bin/activate &&
                         DISPLAY=:0 LIBGL_ALWAYS_SOFTWARE=1
                         PYTHONPATH=<repo>/../pddlstream_lib

         run_eval    ->  cd <repo> && source wsl_env/bin/activate &&
                         PYTHONPATH=<repo>/../pddlstream_lib
                         python3 eval_runner.py $args
         plot_eval   ->  cd <repo> && source wsl_env/bin/activate &&
                         python3 eval_plotter.py $args

       One-time wsl_env bootstrap (wsl_env/ is gitignored):
         wsl bash -lc "cd <repo> && python3 -m venv wsl_env &&
           source wsl_env/bin/activate &&
           pip install pybullet numpy matplotlib"
       matplotlib is required by eval_plotter.py — without it the
       plotter prints text tables instead of PNGs (look for the
       "(matplotlib not available)" marker in stdout).

       plot_eval: pass the /mnt/c/... WSL form of the aggregated.csv
       path.  PowerShell mangles backslashes when a Windows path has
       escape-like substrings (\U, \H, ...).  Three PNGs land next
       to the CSV.

       PowerShell pitfalls:
       - "[plotter] not found: C:Users..." (no backslashes) — a
         Windows path lost its backslashes inside a double-quoted
         "$(...)"; use the /mnt/c/... path or Join-Path as one arg.
       - "(matplotlib not available)" — wsl_env/ lacks matplotlib;
         re-run the bootstrap pip install.
       - "bash: ...conda.sh: No such file or directory" — the shims
         activate the wsl_env venv, not anaconda; install into the
         venv, not base.

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

- Parallel FastDownward / GPU robotics sim — post-thesis research
  direction.  Idea raised 2026-05-10 (later) by user; web-searched
  same day for prior work, findings below.  Bullet refined from
  "speculative" to "mostly-solved on CPU; dead-end on GPU for FD
  specifically; sim-side GPU exists but means a different simulator."

  ON THE FASTDOWNWARD SIDE — substantial prior work exists:
    - HDA* / Distributed Fast Downward (Kishimoto, Fukunaga, Botea —
      ICAPS 2009; arXiv 1201.3204) — Hash-Distributed A* as a drop-in
      replacement for FD's sequential A*.  MPI-based; works on shared
      memory, distributed, or mixed.  30-60x speedup on 128 cores.
      Uses Abstract Zobrist Hashing to assign each state to a unique
      process for local duplicate detection.  Live code:
      github.com/jinnaiyuu/distributed-fast-downward (research code,
      not tracking modern FD; targets OPTIMAL A*, not the satisficing
      greedy best-first that PDDLStream's adaptive algorithm uses —
      so a drop-in port to our pipeline is non-trivial).
    - ArvandHerd (IPC 2011 winner, parallel track) — 4 Arvand configs
      + 1 LAMA config running on separate processors with minimal
      cross-process comms.  This is exactly the "FD portfolio mode"
      angle; it works because the "find a meaningful split" problem
      is hard and config-portfolio is the practical workaround.
    - Fast Downward Stone Soup (Helmert et al., ICAPS 2011) and
      Fast Downward Remix — sequential portfolio configurators
      (15 of 38 FD configs, varying time slices).
    - Springer 2021 "Parallel best-first search algorithms for
      planning problems on multi-core processors" — async work-pool
      design with abstraction-based duplicate-detection partitioning;
      maps directly to a "split by CPU thread count" framing.
    - arXiv 2407.21485 (2024) — Parallel Strategies for Best-First
      Generalized Planning; recent active research.

  ON THE GPU SIDE — essentially dead-end for classical PDDL planning:
    - FD's data structures are pointer-heavy and not trivially
      vectorisable; the literature has very few wins.  The most
      likely win is heuristic batching (e.g. pattern-database
      lookups) — narrow, not a general parallelisation.
    - LLM-generated heuristics on GPU (arXiv 2501.18784, 2025) is
      the more promising angle: don't parallelise the search,
      parallelise the heuristic computation.

  ON THE SIMULATOR SIDE — PyBullet GPU is dormant; alternatives
  exist but mean swapping simulator entirely:
    - Bullet 3.x has an OpenCL GPU rigid-body pipeline (Erwin
      Coumans, 2012) — 100k bodies real-time on a Radeon 7970.
      Never wired into PyBullet; maintainer position on GitHub
      issue #3105 is "may be enabled in the future."  Code dormant
      since AMD stopped funding the work in ~2013.
    - For GPU-accelerated robotics sim today: NVIDIA Isaac Sim /
      Isaac Lab (thousands of envs in parallel, Omniverse-based);
      MuJoCo MJX (JAX); Genesis.  All require a full simulator
      swap — out of thesis scope.
    - Within PyBullet itself, the realistic CPU-parallelism win
      is rayTestBatch numThreads — see #69 for the 6-call-site
      one-line patch.

  WHY THIS STAYS FOR LATER (not promoted to a numbered issue):
    - Even if HDA*/MPI was wired into the planner stack, our
      wall-clock perf complaint is upstream of FD search itself
      (see #50/#62 — Python-side stream/init redundancy).  An
      infinitely-fast FD would shrink an already-small slice.
    - The realistic CPU-side win that doesn't require any
      simulator/planner swap is captured concretely as #69
      (rayTestBatch numThreads).
    - GPU is research-grade work measured in months, not days,
      and the literature is thin enough that it's a dissertation
      topic in its own right — not a pre-submission tweak.

  Cross-references: #50/#62 (Python-side perf hunt — same goal,
  different layer); #69 (rayTestBatch numThreads — the realistic
  spinoff).

- Planner stack (post-thesis / exploratory): move off Fast Downward toward

  integration).  Goal is a swappable backend without investing further in

#78. SENSE SHOULD UPDATE BELIEF FOR EVERY SHADOW THE RAY BUNDLE CROSSES, NOT JUST THE QUERIED ONE
================================================================================
Status:   OPEN.  Raised 2026-05-14 (later) by user during #76
          fix verification.  "sense should sense all shadows,
          not just the one we are looking at."
Priority: TIER 2 — perception efficiency / plan-length
          reduction.  Not a correctness bug; today's pipeline
          works.  The optimisation collapses sequential
          per-shadow senses into one camera observation that
          informs many regions at once, matching how a fixed
          overhead camera actually sees the workspace.
Where:    execution.py sense_shadow_raycasting (line ~44) —
          the rayTestBatch call already returns per-ray
          hit_position + hit_uid; just don't throw away the
          intersections that fall in non-queried shadow AABBs.
          execution.py handle_sense_action — the found_target
          / clear_but_empty / contains_nontarget / still_blocked
          branch dispatch currently fires for one shadow_id;
          needs to fire for each shadow with strong-enough
          per-ray signal.
          PDDL: keep :action sense unchanged.  The multi-region
          update lands in the dispatcher's between-plan registry
          / belief diff (known_empty_shadows + visible_target_-
          locations + refresh_object_aabbs), which the planner
          re-reads via _build_init on every replan.  Cleaner
          than extending the action signature.
Depends:  None directly.  Benefits from #72 [DONE] (sharper
          shadow AABBs after the two-slab carve mean per-ray
          intersection tests are clean) and #71 [DONE] (live
          OBJECT AABBs feed accurate hit classification).
Effort:   1-2 days for the perception change + dispatcher
          plumbing.  No domain change required under option (a).

What:  Today the sense pipeline:
       1. Plan picks one shadow_id.
       2. rayTestBatch from camera → 8 AABB corners + 1 centre
          (audit #69 numThreads=0).
       3. Branch on the SINGLE outcome for the queried shadow.

       But the rays we cast form a bundle from the overhead camera
       down to the target volume; many other shadow AABBs lie
       along these paths.  Each ray's hit position relative to
       every shadow it crosses is observable for free: if the
       first physical hit is past a shadow's exit plane, that
       shadow is empty along that ray.  Across the rays per
       sense × shadows intersected we get a dense per-shadow
       occupancy signal for the cost of one rayTestBatch call.

       Concrete payoff on find-and-tray-stack: typical runs
       sense 3-7 shadows sequentially to find one hidden
       target.  Each sense is move → sense in the PDDL plan
       (~2-4 s of arm motion + camera read).  If a single
       observation informs 3-5 shadows, the expected sense-
       action count per run drops roughly by that factor —
       plan length shrinks, eval wall-clock drops, replan
       count drops.

Fix:   ONE PERCEPTION-LAYER COMMIT.

       Step 1 — extend sense_shadow_raycasting to bin per-ray
       outcomes by EVERY shadow AABB the ray crosses (not just
       the queried one).  For each ray (origin, dir):
         entry_t, exit_t = ray_aabb_intersection(shadow.AABB)
         first_hit_t      = rayTestBatch result
         if first_hit_t > exit_t:           shadow empty along this ray
         elif entry_t <= first_hit_t <= exit_t:
                                            body present; classify the
                                            hit (target / occluder /
                                            non-target / robot)
         else:                              hit pre-entry — irrelevant
       Aggregate per shadow: {target_hits, occluder_hits,
       nontarget_hits, robot_hits, empty_misses}.

       Step 2 — multi-outcome dispatcher in handle_sense_action.
       Fire the existing outcome branches for every shadow with
       strong enough per-ray coverage (start at >= 50 % of rays
       reaching the AABB):
         - target_hits > 0                  → found_target.
         - target_hits == 0, occluder_hits == 0,
           nontarget_hits == 0, coverage >= 50 %
                                            → clear_but_empty.
         - nontarget_hits > 0               → contains_nontarget
                                              (per detected body).
         - coverage < 50 %                  → leave unchanged
                                              (insufficient signal).

       Step 3 — belief / registry diff loop the dispatcher over
       the SET of informed shadows instead of the single shadow_id.
       Each informed shadow goes through the same helpers
       (refresh_object_aabbs, compute_shadow_blockers,
       reboxelize_free_space) used today for one shadow.

Care:
       (1) Coverage threshold matters.  Too low → false-empty
           on a region touched by one edge-grazing ray.  Too
           high → most shadows never get credited and we're
           back to per-region sensing.  Start at 50 %, log the
           per-shadow coverage for tuning.
       (2) found_target uses a HIT, not a miss.  Conservative
           on target detection, permissive on emptiness — the
           right asymmetry.  Don't let a single grazing target
           hit fire a found_target for a barely-observed region.
       (3) shadow_occluder_map gets rebuilt after every sense
           via compute_shadow_blockers.  With many shadows
           flipping view-clear in one action, run the audit
           #58 / #76 paths on seeds 1905790753 + 999 as a
           regression smoke check — both read the current
           registry per-plan so they should ride along
           naturally, but verify.
       (4) Eval impact: audit #73 plot 4 (sense-action count)
           is the direct indicator.  Expect a per-run sense
           count drop on find-and-tray-stack; re-run plot 4
           post-fix.
