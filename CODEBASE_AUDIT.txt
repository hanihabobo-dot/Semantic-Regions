CODEBASE POLICY
================================================================================

GIT COMMITS IN POWERSHELL — DO NOT USE BASH HEREDOCS
       The environment uses Windows PowerShell. Do NOT use bash-style
       heredocs (`cat <<'EOF'`) for git commit messages, as they will hang
       or fail. Use standard `git commit -m "message"` instead.

GIT PUSH — BOTH REMOTES (GITHUB + GITLAB)
       After pushing meaningful work, push the same branch to BOTH hosts
       so they stay in sync:
         git push github <branch>
         git push origin <branch>
       Remotes: `github` → github.com (hanihabobo-dot/pybullet);
                `origin` → git.rwth-aachen.de (hani.alassiri.alhabboub/pybullet).
       Do not treat a single-remote push as "done" unless the other host
       is intentionally out of scope for that change.

COMMENTED-OUT CODE — DO NOT DELETE
       Commented-out code is preserved intentionally as inline
       documentation, quick-enable toggles, and reference for
       re-implementation.
       Rule: NEVER delete without explicit approval.  Move to archive/.

DEAD CODE — PRESERVED FOR FUTURE USE
       Dead methods/imports documented in old #87–#92 (resolved archive)
       are NOT scheduled for deletion.  No runtime cost; may be needed
       for evaluation framework, real perception, etc.

WIKI — UPDATE AFTER EVERY CHANGE
       Project wiki: pybullet.wiki.git (GitHub wiki repo, not main repo).
       Local clone: ../pybullet.wiki/
       Update after any change to architecture, data structures,
       algorithms, PDDL domain, execution pipeline, or design decisions.

ARCHIVE — TRACKED AND PUSHED
       archive/ contains superseded implementations and audit history.
       Tracked by git.  Must NOT contain personal information.

PYBULLET — PLANNING VS. GUI (audit #46 supersedes old #60 full-plan lock)
       PDDLStream planning runs against a separate DIRECT PyBullet client
       (boxel_env.plan_client_id); the GUI window stays interactive during
       long planner.plan() calls.  RenderingLock may still wrap brief IK
       solves on the GUI client during execution (execute_pick / place /
       stack), so a short flicker is possible per action — not a multi-
       minute freeze.  If WSLg/X11 still drops under heavy load, restart
       WSL (wsl --shutdown from PowerShell, then reopen terminal).

WSL VIRTUAL ENVIRONMENT — DO NOT DELETE
       wsl_env/ is the WSL Python venv (gitignored, local only).
       Deleting it breaks the entire pipeline.
       Recreate: python3 -m venv wsl_env && pip install numpy pybullet

BOXEL_FITS / STATIC INIT FACTS — REBUILT EACH REPLAN (old #102)
       Predicates such as (boxel_fits ?o ?b) are precomputed in
       pddlstream_planner._build_init() from the current BoxelRegistry
       snapshot.  Not a PDDLStream test stream — an earlier implementation
       used one (test-boxel-fits), but adaptive search re-evaluated it
       heavily across skeletons and dominated runtime despite a cheap
       per-call check.  stream.pddl no longer declares that stream; domain
       and predicate are unchanged.  As of 2026-04-15, free-space boxels
       ARE re-merged and re-split between replans (reboxelize_free_space
       after sense); the registry is mutated in place before each
       planner.plan() call, and shadow removal / new object creation after
       sense also update the registry.  The planner is stateless between
       calls — all context is passed in each plan() call.

       The 4 drift channels between PDDL symbolic state and PyBullet

           registry size (look for O(N^2) over object/shadow lists).

#58. PRESERVE GRASP ACROSS REPLANS (CURRENTLY DROPS HELD OBJECT)
     [DONE 2026-05-08]
================================================================================
Priority: TIER 2 — Affects every replan that fires while the gripper
          holds an object.  User-flagged 2026-05-08 after observing
          yellow_object dropped mid-air at the PLAN #1 -> PLAN #2
          transition in logs/run_2026-05-08_10-49-01.
Where:    test_full_pipeline.py main() — release_held_object_in_place
          loop at the top of each replan iteration (line ~860; runs
          before planner.plan() to drop the held object).
          pddlstream_planner.py _build_init — currently emits
          ('handempty',) unconditionally (line ~504).
Depends:  None.  Conceptually independent of #50 / #57.
Effort:   <1 day.

What:  When the inner action loop breaks mid-plan (sense returned
       empty, IK failed, missing shadow, etc.) while the gripper is
       still holding an object, the current code RELEASES the object
       in place before the next planner.plan() call.  The release is
       there to keep symbolic state consistent — _build_init
       unconditionally emits ('handempty',), so a plan that started
       with (holding ?o) would let the planner schedule another pick
       and end up with two objects dangling from the EE.  The
       release-then-replan pattern avoids that, but it's wasteful:

       - Loses progress: the next plan must re-pick the dropped
         object before doing anything productive.
       - Drops at uncontrolled poses: the EE position when the loop
         broke (e.g. above the tray after move-to-tray, when sense
         then returns empty), which physics may bounce off the table.
       - Adds ~4 actions per re-grasp cycle (move + pick + move +
         place / stack) to the next plan.
       - Pollutes the registry: the dropped object's new pose may
         intersect free-space boxels, triggering reboxelization
         churn that would otherwise not be needed.

       Concrete example from logs/run_2026-05-08_10-49-01:
         PLAN #1 picked yellow_object then sensed shadow_of_yellow_-
         object empty.  Loop breaks → release_held_object_in_place
         drops yellow at EE pose (-0.181, 0.436, 0.351).  PLAN #2
         then plans pick(red) → place(red, free_005) → re-pick(yellow)
         → ... 13 actions total for what would have been ~9 if yellow
         had stayed in the gripper.

Fix:   ONE COMMIT — propagate (holding ?o) into the next plan's init:
       (a) test_full_pipeline.py main(): instead of releasing on
           replan, capture (held_object_name, grasp) and pass into
           the next planner.plan() call as a new kwarg, e.g.
           held_obj=("yellow_object", grasp_yellow_object_1).
       (b) pddlstream_planner._build_init: when held_obj is set,
           emit ('holding', held_obj_name) and SKIP the
           ('handempty',) emission.  Reuse the existing grasp atom
           if it's already certified, otherwise re-certify via
           sample-grasp + cached output.
       (c) Streams: ensure plan_motion correctly carries the held
           body's ignored_body_ids across the replan boundary
           (already handled via RobotConfig.ignored_body_ids on
           current_config — verify only).
       (d) Keep release_held_object_in_place as a SAFETY NET for
           cases where the held object is in a bad pose for
           continuation (e.g. plan_motion rejected every config
           because the held cube collides with everything).
           Trigger condition: planner returns no plan with held_obj
           set; fall back to release-and-replan.

       Smoke test: re-run --goal find-and-tray-stack on default
       scene with the same seed.  Expect plan_count to drop (PLAN
       #2 becomes a continuation rather than a re-grasp cycle).
