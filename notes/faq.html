What does the IK + waypoint block in execute_pick do? test_full_pipeline.py:675-678

The code in question

pc = env.client_id
approach_joints = solve_ik(robot_id, approach_ee, grasp.orientation, pc)
contact_joints = solve_ik(robot_id, contact_ee, grasp.orientation, pc)
lift_joints    = solve_ik(robot_id, lift_ee,     grasp.orientation, pc)

Line-by-line

pc = env.client_id — grabs the PyBullet physics client ID. PyBullet supports multiple simulation instances; the client ID tells every subsequent API call which instance to talk to.

The three solve_ik() calls compute inverse kinematics for the three key waypoints of a top-down pick motion:

VariableTargetPurpose
approach_joints approach_ee — 10 cm above the object Pre-grasp hover; clears scene objects while descending.
contact_joints contact_ee — the actual grasp point Where the gripper makes contact and closes.
lift_joints lift_ee — 25 cm above the object Post-grasp lift; clears occluders for subsequent moves.

Each call passes the robot body ID, a 3D target position (derived from the object’s current position + grasp offset + vertical offset), the desired end-effector orientation, and the physics client ID.

solve_ik (in robot_utils.py:235) uses null-space IK with a rest-pose seed: it temporarily resets the arm to a known rest configuration, solves IK, validates joint limits, and restores the original state. It returns a 7-element joint-angle array, or None if no valid solution exists.

The subsequent code handles failures — aborting if contact IK fails, falling back gracefully for approach/lift — then executes the pick sequence: approach → open gripper → contact → close gripper → attach → lift.

Q2

What is inverse kinematics and how does solve_ik work? robot_utils.py:235-312

+

The core problem

The Panda arm has 7 joints (shoulder, elbow, wrist, etc.). Together, their angles determine where the gripper ends up in 3D space.

  • Forward kinematics is straightforward: given 7 joint angles → compute gripper position. Pure geometry.
  • Inverse kinematics (IK) is the reverse: given a desired gripper position + orientation → compute the 7 joint angles that reach it.

IK is harder because there may be multiple solutions, no solution (out of reach), or — with 7 joints for 6 DOF — a redundant degree of freedom (the “null space”).

Step 1 — Save and reset to a known pose

The arm is temporarily set to REST_POSES = [0, -0.785, 0, -2.356, 0, 1.571, 0.785] (a neutral elbows-down configuration). This gives PyBullet’s iterative solver a deterministic starting point. Without it, the solution would depend on the arm’s current position, producing inconsistent results.

Step 2 — Call PyBullet’s IK solver

joint_positions = p.calculateInverseKinematics(
    robot_id, END_EFFECTOR_LINK,
    target_pos.tolist(), orn_list,
    lowerLimits=JOINT_LIMITS_LOW.tolist(),
    upperLimits=JOINT_LIMITS_HIGH.tolist(),
    jointRanges=JOINT_RANGES.tolist(),
    restPoses=REST_POSES,
    maxNumIterations=100,
    residualThreshold=1e-4,
    physicsClientId=physics_client,
)

PyBullet runs up to 100 iterations of a numerical optimizer, adjusting joint angles to minimize the distance between the current gripper position and the target. The null-space bias (restPoses) prefers solutions closest to the rest pose, avoiding contorted configurations.

Step 3 — Validate the result

The solver can slightly overshoot physical joint limits. Solutions within 0.1 rad (~5.7°) are clipped to valid range; anything beyond is rejected (return None).

Step 4 — Restore original arm state

The arm is put back to where it was. The solve was a mathematical query — it must not physically move the robot as a side effect.

The output is either a numpy array of 7 joint angles (radians), or None if unreachable. Those angles are then fed to move_robot_smooth() to physically drive the arm.
Q3

What is move_robot_smooth(), are there other move functions, and what is hardcoded? robot_utils.py:314-339

+

What it does

move_robot_smooth() is a low-level joint-space interpolator. It drives the arm from its current joint angles to a target set of 7 angles by:

  1. Reading the current 7 joint angles
  2. Linearly interpolating over steps increments (alpha from 1/steps to 1.0)
  3. Commanding all 7 joints to the interpolated position via PyBullet’s position controller
  4. Stepping the physics simulation once per increment
  5. Optionally sleeping for real-time visualization (120 Hz)

It’s a “go from A to B in a straight line through joint space” primitive — no collision awareness, no path planning.

Hardcoded values

ValueWhereMeaning
range(7)lines 327, 332Number of arm joints — Panda-specific (7-DOF)
240 N·mline 336Motor force — Panda’s peak joint torque
60line 315 (default)Interpolation steps — 0.25 s at 240 Hz sim rate
1/120line 339GUI sleep interval — 120 FPS display rate

The 7 and 240 tie this function to the Franka Emika Panda. Swapping in a different robot would require changing both. See CODEBASE_AUDIT #97 for the broader coupling discussion.

Two levels of motion in the codebase

1. move_robot_smooth() — execution-time primitive (robot_utils.py)

The fast, straight-line interpolator described above. Knows nothing about obstacles. Executes one segment of motion.

2. plan_motion() — planning-time path planner (streams.py:483)

The smart motion function. A PDDLStream stream that:

  • Tries a direct (linear) path and checks for collisions
  • If blocked, runs bidirectional RRT-Connect to find a collision-free path
  • Applies shortcut smoothing (75 iterations)
  • Outputs a Trajectory — an ordered list of RobotConfig waypoints

How they work together

plan_motion()         → produces Trajectory (list of waypoints)
      ↓
test_full_pipeline.py → iterates over traj.waypoints
      ↓
move_robot_smooth()   → executes ONE waypoint-to-waypoint segment

Pick/place functions skip plan_motion entirely — they compute IK waypoints directly and feed them to move_robot_smooth(), relying on approach/lift heights to stay clear of obstacles.

Removed functions

move_robot_to_pos() and compute_ik() were removed in favour of the cleaner separation: solve_ik() for computing joint angles, move_robot_smooth() for executing them. See CODEBASE_AUDIT resolved items.

Q4

Why don’t pick and place update at_config in the PDDL model? domain_pddlstream.pddl:169-176

+

Short answer

No — this is not standard practice. In a textbook PDDL TAMP system you would normally update at_config after every action that moves the robot. This is a deliberate simplification with real trade-offs.

What “correct” PDDL would look like

A faithful model would add a post-action config parameter:

(:action pick
  :parameters (?o ?b ?g ?q ?q_post)
  :precondition (and
    (at_config ?q)
    (kin_solution ?o ?b ?g ?q)
    ;; ...
  )
  :effect (and
    (holding ?o)
    (not (handempty))
    (not (obj_at_boxel ?o ?b))
    (at_config ?q_post)       ;; where the robot ACTUALLY ends up
    (not (at_config ?q))      ;; no longer at the pick config
  ))

This would require an additional stream to compute ?q_post (the lift configuration).

Why this project doesn’t do that

Two safety nets make it work in practice:

  1. Execution-side tracking — after every action, Python reads the robot’s actual joint state from PyBullet (p.getJointState) and stores it in current_config. On replan, this real config is passed to the planner.
  2. Replanning frequency — the reactive loop replans after almost every sense action (and on IK failures), so the stale at_config rarely persists long enough to cause problems.

When it CAN break

If the planner chains pick → move in a single plan, the move action’s precondition (at_config ?q_pick) is symbolically satisfied, but the robot is physically at post_pick_lift. The trajectory was planned from the wrong starting point, causing:

  1. Arm jerk — the position controller jumps to catch up with waypoints planned from a different config
  2. Invalid collision checks — plan_motion validated the path from q_pick, but the actual starting config was never checked
  3. Physical collisions — the trajectory may pass through obstacles when starting from the true post-action config

The proper fix

Two approaches are documented in CODEBASE_AUDIT #62, #95:

  1. Decompose pick/place into sub-actions — approach, grasp, lift, each with their own at_config updates and streams. Textbook-correct but increases planning complexity.
  2. Execution-time state reconciliation — before each action, re-derive configs from the actual physical state rather than trusting the plan’s symbolic parameters.
This is a documented architectural debt, not something you’d normally accept in a production TAMP system. It works here because replanning catches the drift before it accumulates. See CODEBASE_AUDIT #62, #95(a)(i)-(iii) and domain_pddlstream.pddl lines 169–181.
